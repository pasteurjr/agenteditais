# Caso de Uso: UC-001

## Nome
Login e Autenticacao

## Objetivo
Permitir que o usuario acesse o sistema Agente de Editais informando suas credenciais (email e senha), obtendo um token JWT que autoriza o acesso a todas as funcionalidades protegidas.

## Escopo
Modulo de Autenticacao — LoginPage.tsx / Backend Auth

## Atores
- Ator primario: Usuario do sistema (operador comercial, gestor de licitacoes)

## Gatilho
O usuario acessa a URL raiz do sistema e visualiza a tela de login.

## Pre-condicoes
1. O sistema (frontend e backend) esta operacional
2. O usuario possui uma conta cadastrada com email e senha validos
3. O banco de dados esta acessivel

## Pos-condicoes
1. O usuario possui um access_token JWT armazenado em localStorage("editais_ia_access_token")
2. O usuario possui um refresh_token para renovacao automatica
3. O usuario e redirecionado para o Dashboard
4. Todas as requisicoes subsequentes incluem o header Authorization: Bearer {token}

## Regras de negocio
- RN-AUTH-001: Access token tem validade curta; refresh token dura 30 dias
- RN-AUTH-002: Usuario pode estar vinculado a multiplas empresas (switch de empresa)
- RN-AUTH-003: get_current_user_id() extrai user_id do JWT para escopo de dados
- RN-AUTH-004: Decorator @require_auth protege todos os endpoints autenticados
- RN-AUTH-005: Credenciais invalidas retornam mensagem de erro sem revelar qual campo esta incorreto

## Componentes de UI envolvidos
- LoginPage (pagina completa com formulario)
- Icone FileText (lucide-react, 48px) como logotipo
- Titulo "Agente Editais"
- Subtitulo "Assistente para analise de editais e licitacoes publicas"
- Campo de input: email (type="email")
- Campo de input: senha (type="password")
- Botao de submit: "Entrar"
- Div de mensagem de erro (login-error)
- Link "Criar conta" para alternar para RegisterPage
- Botao "Login com Google" (placeholder, ainda nao implementado)

## Endpoints ou servicos envolvidos
- POST /api/auth/login — autentica usuario, retorna access_token + refresh_token
- POST /api/auth/register — cria novo usuario (fluxo alternativo)
- POST /api/auth/refresh — renova access_token com refresh_token
- POST /api/auth/logout — invalida sessao

## Dados manipulados
- Entrada: email (string), senha (string)
- Saida: access_token (JWT string), refresh_token (JWT string)
- Armazenamento local: localStorage("editais_ia_access_token")

## Fluxo principal

### Passo 1: Acesso a tela de login
- **Acao:** O ator abre o navegador e acessa a URL do sistema (http://localhost:5175)
- **Elemento UI:** Pagina LoginPage renderizada com formulario centralizado
- **Resposta visual:** O sistema exibe a tela de login com icone, titulo "Agente Editais", subtitulo, campos email e senha vazios, botao "Entrar" e link "Criar conta"
- **Efeito funcional:** Nenhuma requisicao de rede neste passo; o frontend verifica se ja existe token valido em localStorage
- **Requisicao:** Nenhuma
- **Persistencia:** Nenhuma
- **Evidencia:** Screenshot da tela de login completa (runtime/screenshots/UC-001/01_tela_login.png)

### Passo 2: Preenchimento do email
- **Acao:** O ator clica no campo "Email" e digita seu endereco de email
- **Elemento UI:** Input text com label "Email" (id="email")
- **Resposta visual:** O campo exibe o texto digitado; nenhum feedback adicional
- **Efeito funcional:** Estado React "email" e atualizado via setEmail(e.target.value)
- **Requisicao:** Nenhuma
- **Persistencia:** Apenas estado local do componente
- **Evidencia:** Screenshot com campo email preenchido (runtime/screenshots/UC-001/02_email_preenchido.png)

### Passo 3: Preenchimento da senha
- **Acao:** O ator clica no campo "Senha" e digita sua senha
- **Elemento UI:** Input password com label "Senha" (id="password")
- **Resposta visual:** O campo exibe caracteres mascarados
- **Efeito funcional:** Estado React "password" e atualizado via setPassword(e.target.value)
- **Requisicao:** Nenhuma
- **Persistencia:** Apenas estado local do componente
- **Evidencia:** Screenshot com campo senha preenchido (runtime/screenshots/UC-001/03_senha_preenchida.png)

### Passo 4: Submissao do formulario
- **Acao:** O ator clica no botao "Entrar" ou pressiona Enter
- **Elemento UI:** Botao submit "Entrar"
- **Resposta visual:** O botao entra em estado de loading (isLoading=true), impedindo duplo clique
- **Efeito funcional:** handleSubmit e invocado; setError("") limpa erros anteriores; setIsLoading(true) ativa estado de carregamento
- **Requisicao:** POST /api/auth/login com body { email, password }
- **Persistencia:** Nenhuma neste passo (aguardando resposta)
- **Evidencia:** Screenshot do estado de loading (runtime/screenshots/UC-001/04_loading.png)

### Passo 5: Resposta do backend (sucesso)
- **Acao:** Nenhuma do ator — o sistema processa a resposta
- **Elemento UI:** Nenhuma visivel (transicao automatica)
- **Resposta visual:** A tela de login desaparece e o Dashboard e renderizado
- **Efeito funcional:** O AuthContext recebe access_token e refresh_token; armazena em localStorage; atualiza estado de autenticacao; redirect para Dashboard
- **Requisicao:** Resposta 200 do POST /api/auth/login com { access_token, refresh_token }
- **Persistencia:** access_token salvo em localStorage("editais_ia_access_token"); refresh_token salvo em localStorage
- **Evidencia:** Screenshot do Dashboard apos login (runtime/screenshots/UC-001/05_dashboard_apos_login.png)

## Fluxos alternativos

### FA-001: Registro de novo usuario
1. O ator clica no link "Criar conta"
2. O sistema exibe a pagina RegisterPage com campos nome, email, senha e confirmacao de senha
3. O ator preenche os campos e submete o formulario
4. POST /api/auth/register e invocado
5. Se sucesso, o usuario e criado e pode retornar ao login
6. Se erro, mensagem de validacao e exibida

### FA-002: Login com Google (placeholder)
1. O ator clica no botao "Login com Google"
2. O sistema exibe alert("Login com Google sera implementado em breve!")
3. Nenhuma acao adicional e executada

## Fluxos de excecao

### FE-001: Credenciais invalidas
1. O ator submete email ou senha incorretos
2. Backend retorna erro de autenticacao
3. O catch no handleSubmit captura o erro
4. setError() exibe a mensagem na div "login-error" (texto vermelho acima do formulario)
5. isLoading retorna a false
6. O ator pode tentar novamente

### FE-002: Erro de rede / backend indisponivel
1. O ator submete o formulario mas o backend nao responde
2. O catch captura o erro de rede
3. setError("Erro ao fazer login") e exibido
4. isLoading retorna a false

### FE-003: Campos vazios
1. O ator clica em "Entrar" sem preencher email ou senha
2. Validacao HTML5 (required) impede submissao
3. Nenhuma requisicao e feita

## Criterios de validacao
1. A tela de login renderiza com todos os elementos (icone, titulo, campos, botao, link)
2. O botao "Entrar" dispara POST /api/auth/login com email e senha
3. Credenciais validas resultam em redirect para Dashboard
4. access_token e persistido em localStorage
5. Credenciais invalidas exibem mensagem de erro sem redirect
6. Estado de loading impede duplo clique
