# Caso de Uso: UC-011

## Nome
Configuracao de Parametros de Score

## Objetivo
Permitir que o administrador configure os pesos das 6 dimensoes de score de aderencia, defina limiares de decisao GO/NO-GO e parametros comerciais/logisticos que influenciam a validacao de editais.

## Escopo
Modulo Parametrizacoes — aba/secao Score e Comercial

## Atores
- Administrador (operador do sistema com permissao de configuracao)

## Gatilho
O administrador acessa a ParametrizacoesPage e navega para a secao de configuracao de Score.

## Pre-condicoes
1. Usuario autenticado no sistema
2. Backend ativo na porta 5007
3. Tabela parametros-score existente no banco de dados
4. Ao menos um registro de parametros-score cadastrado

## Pos-condicoes
1. Pesos das 6 dimensoes de score atualizados no banco
2. Limiares GO/NO-GO configurados
3. Parametros comerciais (markup, custos, frete) atualizados
4. Proxima validacao de edital usara os novos parametros

## Regras de negocio
- RN-PAR-001: Pesos de score devem somar 100% — distribuicao padrao: tecnico 35%, comercial 10%, documental 15%, complexidade 15%, juridico 20%, logistico 5%
- RN-VAL-001: 6 dimensoes de score — tecnico, comercial, documental, complexidade, juridico, logistico
- RN-VAL-003: Recomendacao automatica baseada no score: >= limiarGo PARTICIPAR, entre limiarNogo e limiarGo AVALIAR, < limiarNogo NAO PARTICIPAR
- RN-UC011-001: Limiares GO/NO-GO sao configurados para score geral (padrao: Go=70, No-Go=40)
- RN-UC011-002: Limiares especificos por dimensao podem ser definidos (tecnico Go=60/No-Go=30, juridico Go=60/No-Go=30)
- RN-UC011-003: Parametros comerciais incluem: markup padrao, custos fixos, frete base
- RN-UC011-004: Parametros de mercado incluem: TAM (Total Addressable Market), SAM (Serviceable Available Market), SOM (Serviceable Obtainable Market)
- RN-UC011-005: Prazo maximo de entrega e frequencia maxima de participacao sao configurados
- RN-UC011-006: Estados de atuacao definem a regiao geografica para calculo do score logistico
- RN-UC011-007: Palavras-chave e NCMs de busca sao usados na captacao automatica
- RN-UC011-008: Tipos de edital (modalidades) selecionados filtram a busca de editais

## Componentes de UI envolvidos
- ParametrizacoesPage.tsx — secao Score / Comercial
- 6 TextInputs para pesos de dimensao: pesoTecnico, pesoComercial, pesoDocumental, pesoComplexidade, pesoJuridico, pesoLogistico
- TextInputs para limiares GO/NO-GO: limiarGo, limiarNogo, limiarTecnicoGo, limiarTecnicoNogo, limiarJuridicoGo, limiarJuridicoNogo
- TextInputs para parametros comerciais: markupPadrao, custosFixos, freteBase
- TextInputs para mercado: tam, sam, som
- TextInput para prazoMaximo
- SelectInput para frequenciaMaxima
- Checkboxes para estadosSelecionados (27 estados brasileiros + todoBrasil)
- TextArea editavel para palavrasText (palavras-chave de busca)
- TextArea editavel para ncmsText (NCMs de busca)
- Checkboxes para tiposEditalSelecionados (modalidades)
- ActionButton "Salvar" para persistir configuracao
- Info tooltip (showPortfolioHint) com orientacoes

## Endpoints ou servicos envolvidos
- GET /api/crud/parametros-score — carrega parametros de score existentes (loadParamScore)
- POST /api/crud/parametros-score — cria registro de parametros (se nao existir)
- PUT /api/crud/parametros-score/{id} — atualiza parametros existentes
- GET /api/crud/modalidades-licitacao?limit=100 — lista modalidades disponiveis
- GET /api/crud/parametros-score — lista parametros com pesos por dimensao (loadParametros)

## Dados manipulados
- ParametroScoreAPI: id, estados_atuacao (string[]), prazo_maximo (number), frequencia_maxima (string), tam (string), sam (string), som (string), tipos_edital (string[]), palavras_chave (string[]), ncms_busca (string[])
- Pesos de dimensao: pesoTecnico (0.35), pesoComercial (0.10), pesoDocumental (0.15), pesoComplexidade (0.15), pesoJuridico (0.20), pesoLogistico (0.05)
- Limiares GO/NO-GO: limiarGo (70), limiarNogo (40), limiarTecnicoGo (60), limiarTecnicoNogo (30), limiarJuridicoGo (60), limiarJuridicoNogo (30)
- Parametros comerciais: markupPadrao, custosFixos, freteBase
- Mercado: TAM, SAM, SOM
- Modalidades: lista de { id, nome, ativo }

## Fluxo principal

### Passo 1: Acessar secao Score na Parametrizacoes
**Acao do ator:** O administrador acessa a ParametrizacoesPage e navega ate a secao de Score/Comercial.
**Elemento UI:** Card com titulo "Pesos de Score" ou secao correspondente com icones DollarSign e Target.
**Resposta visual:** Campos de pesos das 6 dimensoes exibidos com valores atuais. Limiares GO/NO-GO visiveis. Parametros comerciais exibidos.
**Efeito funcional:** loadParamScore carrega parametros do backend. Valores preenchidos nos campos de estado.
**Requisicao de rede:** GET /api/crud/parametros-score.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot da secao Score com pesos e limiares visiveis.

### Passo 2: Ajustar pesos de dimensao
**Acao do ator:** O administrador altera os pesos das dimensoes de score (ex: aumentar peso tecnico de 35% para 40%).
**Elemento UI:** TextInputs para cada dimensao: Tecnico, Comercial, Documental, Complexidade, Juridico, Logistico. Valores decimais (0.35, 0.10, etc.).
**Resposta visual:** Valores nos campos mudam conforme digitacao. Sistema pode exibir soma total dos pesos para validacao visual.
**Efeito funcional:** Estados pesoTecnico, pesoComercial, etc. atualizados.
**Requisicao de rede:** Nenhuma (alteracao local).
**Persistencia:** Nenhuma (ainda nao salvo).
**Evidencia esperada:** Screenshot com pesos alterados.

### Passo 3: Configurar limiares GO/NO-GO
**Acao do ator:** O administrador ajusta os limiares de decisao (ex: Go=75, No-Go=45).
**Elemento UI:** TextInputs para limiarGo, limiarNogo, limiarTecnicoGo, limiarTecnicoNogo, limiarJuridicoGo, limiarJuridicoNogo.
**Resposta visual:** Valores nos campos atualizados. Sistema pode exibir preview da regra (">= 75: PARTICIPAR, 45-74: AVALIAR, < 45: NAO PARTICIPAR").
**Efeito funcional:** Estados de limiares atualizados.
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot com limiares configurados.

### Passo 4: Definir parametros comerciais
**Acao do ator:** O administrador preenche markup padrao, custos fixos e frete base.
**Elemento UI:** TextInputs para markupPadrao, custosFixos, freteBase.
**Resposta visual:** Campos preenchidos com valores monetarios/percentuais.
**Efeito funcional:** Estados de parametros comerciais atualizados.
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot com parametros comerciais preenchidos.

### Passo 5: Selecionar estados de atuacao
**Acao do ator:** O administrador marca os estados onde a empresa atua (checkboxes) ou seleciona "Todo Brasil".
**Elemento UI:** Grid de checkboxes com 27 estados brasileiros (ESTADOS_BR) + checkbox "Todo Brasil" (todoBrasil).
**Resposta visual:** Checkboxes marcados ficam em destaque. Selecionar "Todo Brasil" marca todos os estados.
**Efeito funcional:** Estados estadosSelecionados atualizados (Set). todoBrasil controla selecao global.
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot com estados selecionados no mapa/grid.

### Passo 6: Configurar palavras-chave e NCMs
**Acao do ator:** O administrador edita as palavras-chave e NCMs de busca.
**Elemento UI:** TextArea para palavrasText (modo edicao ativado por editingPalavras). TextArea para ncmsText (modo edicao ativado por editingNcms).
**Resposta visual:** Areas de texto editaveis com listas de termos separados por virgula ou quebra de linha.
**Efeito funcional:** Estados palavrasText e ncmsText atualizados.
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot com palavras-chave e NCMs editados.

### Passo 7: Salvar configuracao
**Acao do ator:** O administrador clica no botao "Salvar" para persistir todas as alteracoes.
**Elemento UI:** ActionButton "Salvar" com icone CheckCircle. Loader exibido durante salvamento (savingParamScore).
**Resposta visual:** Loader aparece durante salvamento. Mensagem de sucesso apos persistencia. Campos voltam a modo visualizacao.
**Efeito funcional:** crudUpdate("parametros-score", paramScoreId, { ... }) chamado com todos os parametros. Se paramScoreId nao existir, crudCreate e usado.
**Requisicao de rede:** PUT /api/crud/parametros-score/{id} com payload completo (pesos, limiares, estados, palavras-chave, NCMs, comercial, mercado, modalidades).
**Persistencia:** Todos os parametros persistidos no banco de dados.
**Evidencia esperada:** Screenshot com mensagem de sucesso apos salvamento.

## Fluxos alternativos

### FA-001: Primeiro acesso (sem parametros existentes)
**Condicao:** Nao existe registro em parametros-score.
**Variacao:** Campos exibidos com valores padrao (35%, 10%, 15%, 15%, 20%, 5%). Ao salvar, crudCreate e usado em vez de crudUpdate.

### FA-002: Selecionar tipos de edital
**Condicao:** Administrador quer filtrar tipos/modalidades de edital.
**Variacao:** Secao de modalidades exibida com checkboxes carregados de /api/crud/modalidades-licitacao. Modalidades selecionadas sao salvas em tipos_edital[].

### FA-003: Definir parametros de mercado (TAM/SAM/SOM)
**Condicao:** Administrador preenche dados de mercado.
**Variacao:** Campos TAM, SAM e SOM preenchidos com valores monetarios. Usados para dimensionamento de oportunidade na validacao.

## Fluxos de excecao

### FE-001: Pesos nao somam 100%
**Condicao:** Soma dos 6 pesos e diferente de 1.0 (100%).
**Resposta:** Sistema pode alertar visualmente (nao ha validacao bloqueante no frontend). Backend pode aceitar qualquer soma.

### FE-002: Erro ao salvar
**Condicao:** PUT /api/crud/parametros-score retorna erro.
**Resposta:** Mensagem de erro exibida. Campos permanecem editaveis para nova tentativa. savingParamScore volta a false.

### FE-003: Erro ao carregar parametros
**Condicao:** GET /api/crud/parametros-score retorna erro.
**Resposta:** Campos exibidos com valores padrao do estado React. loadingParamScore volta a false.

## Criterios de validacao
1. Pesos das 6 dimensoes devem ser carregados do backend e editaveis
2. Limiares GO/NO-GO devem ser configurados (geral + por dimensao)
3. Selecao de estados deve funcionar com checkboxes individuais e "Todo Brasil"
4. Palavras-chave e NCMs devem ser editaveis em modo texto
5. Salvar deve persistir via PUT /api/crud/parametros-score/{id}
6. Valores salvos devem ser utilizados na validacao de editais (ValidacaoPage)
7. Parametros comerciais (markup, custos, frete) devem ser persistidos
