# Caso de Uso: UC-015

## Nome
Salvar editais selecionados

## Objetivo
Permitir ao usuario persistir no banco de dados os editais selecionados na CaptacaoPage, tornando-os disponiveis para analise nas etapas seguintes (Validacao, Precificacao, Proposta).

## Escopo
Modulo Captacao — CaptacaoPage.tsx

## Atores
- Usuario autenticado (Analista Comercial)
- Sistema backend (Flask)
- Banco de dados (SQLite/PostgreSQL)

## Gatilho
Usuario clica no botao "Salvar Selecionados" apos marcar ao menos um edital via checkbox na DataTable.

## Pre-condicoes
1. Usuario esta autenticado no sistema
2. Busca de editais foi realizada (UC-013)
3. Ao menos um edital esta selecionado via checkbox (UC-014)
4. Backend Flask ativo e banco de dados acessivel
5. Empresa do usuario cadastrada no sistema
6. Editais selecionados possuem dados minimos: numero, orgao, objeto
7. Edital nao foi previamente salvo pelo mesmo usuario (verificacao de duplicidade)

## Pos-condicoes
1. Editais salvos no banco de dados com status "novo"
2. Editais aparecem na lista de ValidacaoPage para analise
3. Dados enriquecidos (itens, valores, PDF) persistidos junto ao edital
4. Notificacao de sucesso exibida ao usuario ("X editais salvos com sucesso")
5. Checkbox dos editais salvos e desmarcado apos salvamento
6. Contagem de editais salvos atualizada no Dashboard
7. Edital salvo recebe editalSalvoId para futuras operacoes de update

## Regras de negocio
1. **RN-SAL-001:** CaptacaoPage usa crudCreate para persistir cada edital selecionado
2. **RN-SAL-002:** Verificacao de duplicidade por numero do edital + CNPJ do orgao antes de salvar
3. **RN-SAL-003:** Status inicial do edital salvo e "novo" (visivel na ValidacaoPage)
4. **RN-SAL-004:** Dados PNCP (cnpj_orgao, ano_compra, seq_compra) sao persistidos para referencia futura
5. **RN-SAL-005:** Itens do edital (se disponiveis) sao salvos em tabela associada
6. **RN-SAL-006:** Vinculo com classe/subclasse de produto (classe_produto_id, subclasse_produto_id) e persistido
7. **RN-SAL-007:** Salvamento em lote — multiplos editais salvos em uma unica operacao

## Componentes de UI envolvidos
1. Botao "Salvar Selecionados" (ActionButton com icone Save)
2. Checkbox de selecao na DataTable (estado `selected` de cada EditalBusca)
3. Indicador de contagem de editais selecionados
4. Toast/notificacao de sucesso ou erro apos salvamento
5. Indicador de loading no botao durante operacao de salvamento
6. Modal de confirmacao para salvamento em lote (quando mais de 10 editais)
7. Badge de status "Salvo" nos editais ja persistidos na DataTable

## Endpoints ou servicos envolvidos
1. **POST /api/editais/salvar** — salva edital selecionado no banco
2. **crudCreate('editais')** — operacao CRUD generica para criar registro
3. **crudUpdate('editais')** — atualiza edital existente (se ja salvo previamente)
4. **crudList('editais')** — verifica duplicidade antes de salvar
5. **POST /api/editais/{id}/itens** — salva itens associados ao edital
6. **GET /api/editais/{id}/pdf** — download do PDF para armazenamento local
7. **Transacao de banco:** operacao atomica para salvar edital + itens + vinculo produto

## Dados manipulados
1. **EditalBusca completo:** todos os campos da interface (numero, orgao, uf, objeto, valor, etc.)
2. **Itens do edital:** numero_item, descricao, quantidade, unidade_medida, valor_unitario_estimado
3. **Classificacao:** tipoProdutoInferido, origemInferida, classe_produto_id, subclasse_produto_id
4. **Estrategia:** potencialGanho, intencaoEstrategica, margemExpectativa
5. **Scores iniciais:** score de aderencia (tecnico, comercial, recomendacao)
6. **Metadados PNCP:** cnpj_orgao, ano_compra, seq_compra, url do portal
7. **IDs gerados:** editalSalvoId e estrategiaId retornados pelo backend apos persistencia

## Fluxo principal
1. Usuario revisa editais selecionados (checkboxes marcados) na DataTable da CaptacaoPage
2. Usuario clica no botao "Salvar Selecionados" que fica habilitado quando ha selecao ativa
3. Frontend coleta todos os EditalBusca com selected=true e envia array ao backend via crudCreate
4. Backend verifica duplicidade para cada edital (numero + cnpj_orgao) e descarta duplicados com aviso
5. Backend persiste cada edital novo no banco com status "novo", incluindo itens e classificacao
6. Backend retorna lista de IDs gerados (editalSalvoId) e contagem de salvos vs duplicados
7. Frontend exibe toast de sucesso ("5 editais salvos, 2 ja existiam"), desmarca checkboxes e atualiza badges

## Fluxos alternativos
1. **FA-01 — Salvar edital individual:** Usuario clica no icone Save na linha especifica da DataTable em vez de usar selecao em lote
2. **FA-02 — Edital ja salvo:** Sistema detecta duplicidade e exibe aviso "Edital ja salvo anteriormente" sem criar duplicata
3. **FA-03 — Atualizar edital existente:** Se edital ja existe no banco, sistema oferece opcao de atualizar dados (crudUpdate)
4. **FA-04 — Salvar com estrategia:** Usuario define intencaoEstrategica antes de salvar (estrategico, defensivo, etc.)
5. **FA-05 — Download PDF simultaneo:** Ao salvar, sistema dispara download do PDF do edital em background
6. **FA-06 — Vincular classe de produto:** Usuario associa classe/subclasse antes de salvar para categorizacao
7. **FA-07 — Cancelar salvamento:** Usuario clica em "Cancelar" no modal de confirmacao de lote

## Fluxos de excecao
1. **FE-01 — Nenhum edital selecionado:** Botao "Salvar" desabilitado; se forcado, frontend exibe alerta "Selecione ao menos um edital"
2. **FE-02 — Erro de banco de dados:** Backend retorna erro 500; frontend exibe toast de erro com botao "Tentar novamente"
3. **FE-03 — Timeout de salvamento:** Operacao excede 30s; frontend exibe mensagem de timeout e orienta tentar com menos editais
4. **FE-04 — Dados incompletos:** Edital sem campo obrigatorio (numero ou orgao); backend rejeita com HTTP 422 e detalha campos faltantes
5. **FE-05 — Sessao expirada:** Token JWT invalido; sistema redireciona para login e perde selecao (usuario deve refazer busca)
6. **FE-06 — Conflito de concorrencia:** Outro usuario salva mesmo edital simultaneamente; sistema aplica ultima escrita e informa
7. **FE-07 — Espaco em disco:** Banco de dados sem espaco; backend retorna erro 507 e frontend exibe mensagem de contato com suporte

## Criterios de validacao
1. Botao "Salvar" fica habilitado apenas quando ha editais selecionados
2. Editais salvos aparecem na ValidacaoPage com status "novo"
3. Duplicidade e detectada corretamente (mesmo numero + cnpj nao gera duplicata)
4. Toast de sucesso exibe contagem correta de editais salvos
5. Itens do edital sao persistidos junto ao edital principal
6. Apos salvamento, checkboxes sao desmarcados automaticamente
7. IDs gerados (editalSalvoId) sao retornados e armazenados no estado do frontend
