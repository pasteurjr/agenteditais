# Caso de Uso: UC-053

## Nome
Controlar saldo ARP e caronas

## Objetivo
Permitir ao usuario controlar o saldo de itens de Ata de Registro de Preco (ARP), registrar solicitacoes de carona com validacao automatica de limites legais (Lei 14.133 Art. 82-86): 50% por orgao individual e 2x o total registrado como limite global de caronas.

## Escopo
Modulo Atas — AtasPage.tsx > Aba Saldo ARP

## Atores
- Usuario autenticado (Analista Comercial / Gestor de ARP)
- Sistema backend (Flask)
- Motor de validacao legal (Art. 82-86 Lei 14.133)
- Orgaos solicitantes de carona (externos)

## Gatilho
Usuario seleciona uma ata na lista de "Minhas Atas" e acessa a aba "Saldo ARP" para consultar saldos e registrar caronas.

## Pre-condicoes
1. Usuario esta autenticado no sistema
2. Ata salva e selecionada com itens extraidos (UC-045 + UC-046)
3. Backend Flask ativo com endpoints de saldo ARP
4. Itens da ata com quantidade_registrada, consumido_participante e consumido_carona preenchidos
5. Regras de limite de carona (Art. 82-86) implementadas no backend
6. AtasPage acessivel no menu do sistema
7. Ata vigente (data_vigencia_fim > hoje) para novas caronas

## Pos-condicoes
1. Saldo disponivel por item calculado e exibido (registrada - consumido_participante - consumido_carona)
2. Carona registrada com orgao solicitante, CNPJ, quantidade e justificativa
3. Validacao de limite individual (50%) aplicada no backend
4. Validacao de limite global (2x) aplicada no backend
5. Saldo atualizado apos carona aprovada
6. Historico de caronas por item mantido
7. Alerta exibido se carona excede limite

## Regras de negocio
1. **RN-ATA-001:** Limite individual de carona = 50% da quantidade_registrada por orgao solicitante (Art. 82-86)
2. **RN-ATA-002:** Limite global de carona = 2x a quantidade_registrada total para todos os orgaos
3. **RN-ATA-003:** Validacao server-side no endpoint POST de caronas (rejeita se exceder limites)
4. **RN-ARP-001:** saldo_disponivel = quantidade_registrada - consumido_participante - consumido_carona
5. **RN-ARP-002:** Carona requer: orgao_solicitante, cnpj_solicitante, quantidade_solicitada, justificativa
6. **RN-ARP-003:** Carona possui status: pendente, aprovada, rejeitada
7. **RN-ARP-004:** Alertas visuais quando saldo disponivel < 20% do registrado

## Componentes de UI envolvidos
1. TabPanel aba "Saldo ARP" na AtasPage
2. SelectInput para selecao de ata (dropdown com atas salvas)
3. DataTable de saldos: item_descricao, quantidade_registrada, consumido_participante, consumido_carona, saldo_disponivel, valor_unitario
4. ActionButton "Nova Carona" por item com icone Plus
5. Modal de carona: orgao_solicitante, cnpj_solicitante, quantidade_solicitada, justificativa
6. DataTable de caronas por item: orgao, quantidade, status, data
7. Formatacao monetaria R$ para valor_unitario e barra visual de consumo

## Endpoints ou servicos envolvidos
1. **GET /api/atas/{id}/saldos** — retorna saldos por item da ata selecionada
2. **GET /api/atas/{id}/saldos/{saldo_id}/caronas** — lista caronas de um item especifico
3. **POST /api/atas/{id}/saldos/{saldo_id}/caronas** — registra solicitacao de carona com validacao de limites
4. **Validacao backend:** verifica limite 50% individual e 2x global antes de aceitar carona
5. **GET /api/atas/{id}** — dados da ata (vigencia)
6. **GET /api/atas/minhas** — lista de atas para SelectInput
7. **Auth middleware** — validacao de token JWT

## Dados manipulados
1. **ARPSaldo:** id, item_descricao, quantidade_registrada, consumido_participante, consumido_carona, saldo_disponivel, valor_unitario
2. **Carona:** id, orgao_solicitante, quantidade_solicitada, status, data_solicitacao
3. **Payload carona:** orgao_solicitante, cnpj_solicitante, quantidade_solicitada (float), justificativa
4. **Validacao 50%:** quantidade_solicitada <= quantidade_registrada * 0.50 (por orgao)
5. **Validacao 2x:** total_caronas_global <= quantidade_registrada * 2.0
6. **Estado UI:** selectedAta, saldos (ARPSaldo[]), caronas (Carona[]), showCaronaModal
7. **Erro de validacao:** alert(err.error || "Erro ao solicitar carona") quando limites excedidos

## Fluxo principal
1. Usuario acessa aba "Saldo ARP" e seleciona uma ata no SelectInput (dropdown com atas salvas)
2. Frontend chama fetchSaldos(ataId) via GET /api/atas/{id}/saldos e exibe DataTable com saldos por item
3. DataTable mostra: descricao, quantidade_registrada, consumido_participante, consumido_carona, saldo_disponivel, valor_unitario
4. Usuario identifica item com saldo disponivel e clica em "Nova Carona" (icone Plus) que abre Modal
5. Usuario preenche orgao_solicitante, cnpj_solicitante, quantidade_solicitada e justificativa
6. Frontend envia POST /api/atas/{id}/saldos/{saldo_id}/caronas; backend valida limites 50% e 2x
7. Se aprovada, Modal fecha, fetchCaronas e fetchSaldos atualizam; se rejeitada, alert exibe motivo do erro

## Fluxos alternativos
1. **FA-01 — Consultar caronas:** Usuario clica no item para ver historico de caronas via GET /api/atas/{id}/saldos/{saldo_id}/caronas
2. **FA-02 — Saldo esgotado:** Item com saldo_disponivel = 0; botao "Nova Carona" desabilitado ou ausente
3. **FA-03 — Ata vencida:** Data de vigencia expirada; caronas nao permitidas em ata vencida
4. **FA-04 — Sem ata selecionada:** SelectInput vazio; DataTable mostra mensagem de selecionar ata
5. **FA-05 — Limite 50% atingido:** Orgao ja consumiu 50% do item; backend rejeita nova carona desse orgao
6. **FA-06 — Limite global 2x atingido:** Total de caronas = 2x registrado; backend rejeita qualquer nova carona
7. **FA-07 — Carona parcial:** Quantidade solicitada menor que saldo disponivel; carona aprovada parcialmente

## Fluxos de excecao
1. **FE-01 — Limite excedido:** POST retorna erro { error: "Excede limite de 50% por orgao" }; alert exibe mensagem
2. **FE-02 — Token expirado:** Requisicao retorna 401; sistema redireciona para LoginPage
3. **FE-03 — Quantidade invalida:** parseFloat retorna NaN ou <= 0; validacao frontend pode barrar
4. **FE-04 — Ata nao encontrada:** ID invalido; backend retorna 404
5. **FE-05 — Erro de rede:** fetch falha; try/catch captura e loga
6. **FE-06 — CNPJ invalido:** Backend rejeita CNPJ com formato invalido
7. **FE-07 — Concorrencia:** Dois usuarios solicitam carona simultaneamente; primeiro atualiza saldo, segundo pode exceder

1. Saldo disponivel calculado corretamente (registrada - participante - carona)
2. POST /api/atas/{id}/saldos/{saldo_id}/caronas valida limite de 50% por orgao
3. POST valida limite global de 2x a quantidade registrada
4. Carona aprovada atualiza saldo e aparece no historico
5. Carona rejeitada exibe mensagem de erro via alert
6. DataTable exibe todos os campos de saldo corretamente
7. SelectInput de ata funciona e carrega saldos da ata selecionada
