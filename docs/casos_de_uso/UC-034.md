# Caso de Uso: UC-034

## Nome
Upload de peticao externa

## Objetivo
Permitir ao usuario fazer upload de peticao de impugnacao ou esclarecimento ja elaborada externamente (DOCX, PDF), vinculando-a ao edital correspondente e integrando-a ao fluxo de acompanhamento de peticoes do sistema.

## Escopo
Modulo Impugnacao — ImpugnacaoPage.tsx (aba Peticoes — botao Upload)

## Atores
- Usuario autenticado (Analista Juridico / Comercial)
- Sistema backend (Flask)
- Arquivo externo (PDF ou DOCX da peticao)

## Gatilho
Usuario clica no botao "Upload" (icone Upload) na aba Peticoes da ImpugnacaoPage, abrindo modal de upload.

## Pre-condicoes
1. Usuario esta autenticado com token JWT valido
2. Backend Flask ativo na porta 5007
3. Ao menos um edital salvo no sistema para vinculacao
4. Arquivo da peticao disponivel no computador do usuario (PDF ou DOCX)
5. Arquivo com tamanho dentro do limite aceito pelo backend
6. Aba Peticoes da ImpugnacaoPage acessivel e carregada
7. Conexao de rede estavel para upload do arquivo

## Pos-condicoes
1. Arquivo da peticao armazenado no backend vinculado ao edital
2. Registro da peticao criado na lista de peticoes com status "rascunho"
3. Peticao visivel na DataTable da aba Peticoes
4. Arquivo acessivel para download posterior
5. Edital vinculado registrado (edital_id)
6. Data de criacao registrada automaticamente
7. Peticao disponivel para fluxo de acompanhamento (revisao, envio)

## Regras de negocio
1. **RN-UPL-001:** Formatos aceitos: PDF (.pdf) e DOCX (.docx)
2. **RN-UPL-002:** Peticao criada com status inicial "rascunho" apos upload
3. **RN-UPL-003:** Vinculacao obrigatoria a edital existente via SelectInput (uploadEditalId)
4. **RN-UPL-004:** Arquivo armazenado no backend com referencia ao registro da peticao
5. **RN-UPL-005:** Upload usa estado uploadFile (File) gerenciado pelo input type=file do modal
6. **RN-UPL-006:** Apos upload, lista de peticoes e recarregada via loadPeticoes()
7. **RN-UPL-007:** Peticao externa segue mesmo fluxo de status que peticao gerada internamente

## Componentes de UI envolvidos
1. ActionButton "Upload" (icone Upload) na ActionBar da aba Peticoes
2. Modal showUploadModal com campos de formulario
3. SelectInput para selecao de edital vinculado (uploadEditalId) com opcoes "numero - orgao"
4. Input type="file" para selecao do arquivo (uploadFile: File | null)
5. ActionButton "Enviar" para confirmar upload com estado loading
6. Indicador de progresso durante upload do arquivo
7. DataTable atualizada apos upload mostrando nova peticao na lista

## Endpoints ou servicos envolvidos
1. **POST /api/upload** ou **crudCreate("recursos")** — persiste arquivo e registro da peticao
2. **GET /api/editais** — lista editais para SelectInput (getEditais)
3. **crudList("recursos")** — recarrega lista de peticoes apos upload (loadPeticoes)
4. **localStorage.getItem("editais_ia_access_token")** — token JWT para autenticacao
5. **setUploadFile(File)** — gerencia estado do arquivo selecionado
6. **setUploadEditalId(string)** — gerencia estado do edital vinculado
7. **setShowUploadModal(false)** — fecha modal apos upload bem-sucedido

## Dados manipulados
1. **uploadFile:** File | null — arquivo selecionado pelo usuario para upload
2. **uploadEditalId:** string — UUID do edital vinculado a peticao
3. **Peticao resultante:** id, edital_id, tipo, status "rascunho", data_criacao
4. **FormData:** multipart contendo arquivo e metadados (edital_id)
5. **showUploadModal:** boolean — controle de visibilidade do modal
6. **peticoes:** Peticao[] — lista atualizada apos upload
7. **editalOptions:** {value, label}[] — opcoes do SelectInput de editais

## Fluxo principal
1. Usuario acessa aba Peticoes na ImpugnacaoPage e clica no botao "Upload" (icone Upload) na ActionBar
2. Modal de upload abre com SelectInput de edital e campo de selecao de arquivo (input type="file")
3. Usuario seleciona o edital vinculado no dropdown que exibe "numero - orgao" dos editais salvos
4. Usuario clica no campo de arquivo e seleciona peticao elaborada externamente (PDF ou DOCX) do computador
5. Usuario clica em "Enviar"; frontend envia FormData com arquivo e edital_id ao backend via POST
6. Backend armazena arquivo, cria registro da peticao com status "rascunho" e retorna confirmacao
7. Frontend fecha modal, recarrega lista de peticoes via loadPeticoes() e nova peticao aparece na DataTable

## Fluxos alternativos
1. **FA-01 — Substituir arquivo:** Usuario faz novo upload para peticao existente, substituindo arquivo anterior
2. **FA-02 — Multiplos uploads:** Usuario faz upload de varias peticoes para o mesmo edital sequencialmente
3. **FA-03 — Arrastar e soltar:** Usuario arrasta arquivo diretamente para area do modal (drag-and-drop)
4. **FA-04 — Visualizar antes de enviar:** Usuario clica Eye para pre-visualizar conteudo do arquivo antes do upload
5. **FA-05 — Upload com tipo definido:** Usuario seleciona tipo (esclarecimento/impugnacao) antes de confirmar upload
6. **FA-06 — Cancelar upload:** Usuario fecha modal sem enviar; nenhuma alteracao e persistida
7. **FA-07 — Download posterior:** Usuario baixa arquivo uploadado posteriormente via icone Download na DataTable

## Fluxos de excecao
1. **FE-01 — Nenhum edital selecionado:** Botao Enviar desabilitado se uploadEditalId estiver vazio
2. **FE-02 — Nenhum arquivo selecionado:** Validacao impede envio se uploadFile for null
3. **FE-03 — Formato invalido:** Arquivo com extensao nao suportada; frontend ou backend rejeita com mensagem
4. **FE-04 — Arquivo muito grande:** Backend retorna erro 413; frontend exibe mensagem de limite excedido
5. **FE-05 — Erro de rede durante upload:** fetch lanca excecao; frontend exibe mensagem e mantem modal aberto
6. **FE-06 — Token expirado:** Requisicao retorna 401; frontend redireciona para login
7. **FE-07 — Armazenamento indisponivel:** Backend falha ao salvar arquivo; retorna erro 500 com mensagem
