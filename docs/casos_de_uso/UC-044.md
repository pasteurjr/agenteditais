# Caso de Uso: UC-044

## Nome
Calcular score logistico

## Objetivo
Calcular o score logistico de um edital com base em 4 sub-dimensoes (distancia, historico de entregas, custo de frete, prazo), compondo a dimensao logistica do score de aderencia 6D utilizado na validacao de editais.

## Escopo
Modulo Validacao/API — ValidacaoPage.tsx / Endpoint dedicado

## Atores
- Usuario autenticado (Analista Comercial)
- Sistema backend (Flask)
- Motor de IA (DeepSeek) para inferencia de localizacao e custos
- Servico de geolocalizacao (calculo de distancia)

## Gatilho
Score logistico e calculado automaticamente como parte da validacao 6D (UC-016) ou via chamada direta ao endpoint GET /api/validacao/score-logistico/{id}.

## Pre-condicoes
1. Usuario esta autenticado no sistema
2. Edital salvo com dados de orgao e localizacao (UF, municipio)
3. Empresa cadastrada com endereco e CEP para calculo de distancia
4. Portfolio cadastrado com dados de frete e prazos de entrega
5. Historico de entregas anteriores disponivel para calculo de taxa de sucesso
6. Backend Flask ativo com endpoint de score logistico
7. Parametros de pesos das sub-dimensoes configurados em Parametrizacoes

## Pos-condicoes
1. Score logistico geral (0-100) calculado como media ponderada das 4 sub-dimensoes
2. Sub-scores individuais: distancia (0-100), historico (0-100), frete (0-100), prazo (0-100)
3. Score logistico incorporado na dimensao logistica do score 6D (peso 10%)
4. Detalhamento com justificativa de cada sub-score disponivel
5. Resultado persistido no banco vinculado ao edital
6. Pontos de atencao logisticos identificados e listados
7. Score disponivel para consulta via API e UI

## Regras de negocio
1. **RN-LOG-001:** 4 sub-dimensoes com pesos: distancia (30%), historico entregas (25%), custo frete (25%), prazo (20%)
2. **RN-LOG-002:** Score distancia inversamente proporcional a distancia empresa-orgao (menor distancia = maior score)
3. **RN-LOG-003:** Score historico baseado em taxa de entregas no prazo em licitacoes anteriores do mesmo orgao/regiao
4. **RN-LOG-004:** Score frete compara custo de frete estimado com margem disponivel no preco
5. **RN-LOG-005:** Score prazo avalia viabilidade do prazo de entrega exigido vs capacidade da empresa
6. **RN-LOG-006:** Score logistico geral = distancia*0.30 + historico*0.25 + frete*0.25 + prazo*0.20
7. **RN-LOG-007:** Score logistico compoe peso de 10% no score geral 6D (RN-VAL-001)

## Componentes de UI envolvidos
1. ScoreBar da dimensao "Logistico" no painel de validacao da ValidacaoPage
2. Tooltip ou expansao com detalhamento das 4 sub-dimensoes
3. Card de pontos de atencao logisticos (distancia, prazo, frete)
4. Indicador visual de distancia empresa-orgao (mapa ou texto)
5. Barra de progresso de cada sub-dimensao com valor numerico
6. Badge de risco logistico (alto/medio/baixo) baseado no score geral
7. Botao "Recalcular Logistico" para atualizar apos mudanca de dados

## Endpoints ou servicos envolvidos
1. **GET /api/validacao/score-logistico/{id}** — calcula e retorna score logistico do edital
2. **POST /api/editais/{id}/validar** — validacao 6D que inclui calculo logistico
3. **tool_validar_edital** — tool de IA que analisa dimensoes incluindo logistica
4. **crudList('produtos')** — dados de prazo e frete dos produtos do portfolio
5. **GET /api/empresa** — dados de endereco e localizacao da empresa
6. **GET /api/editais/{id}** — dados do edital incluindo orgao e UF
7. **DeepSeek API** — inferencia de custos e viabilidade logistica

## Dados manipulados
1. **Score logistico geral:** float 0-100 (media ponderada das 4 sub-dimensoes)
2. **Sub-score distancia:** float 0-100 baseado em km empresa-orgao
3. **Sub-score historico:** float 0-100 baseado em % entregas no prazo
4. **Sub-score frete:** float 0-100 baseado em custo estimado vs margem
5. **Sub-score prazo:** float 0-100 baseado em dias exigidos vs capacidade
6. **Dados de entrada:** UF orgao, CEP empresa, prazo exigido, peso/volume produtos
7. **Pontos de atencao:** string[] — riscos logisticos identificados

## Fluxo principal
1. Usuario solicita validacao de edital (UC-016) ou chama diretamente GET /api/validacao/score-logistico/{id}
2. Backend recupera dados do edital (orgao, UF, municipio) e dados da empresa (endereco, CEP, capacidade)
3. Sistema calcula sub-score de distancia baseado na distancia geografica empresa-orgao (inversamente proporcional)
4. Sistema calcula sub-score de historico consultando entregas anteriores na mesma regiao/orgao
5. Sistema calcula sub-score de frete estimando custo de transporte vs margem disponivel no preco proposto
6. Sistema calcula sub-score de prazo comparando prazo exigido pelo edital com capacidade produtiva e logistica da empresa
7. Backend aplica pesos (30/25/25/20), calcula score geral, persiste resultado e retorna JSON com scores e pontos de atencao

## Fluxos alternativos
1. **FA-01 — Sem historico de entregas:** Sub-score historico atribuido como 50 (neutro) com flag "dados insuficientes"
2. **FA-02 — Edital sem localizacao:** Sub-score distancia calculado pela UF do orgao com menor precisao
3. **FA-03 — Empresa sem endereco:** Score logistico inteiro recebe 0 com alerta "endereco da empresa nao cadastrado"
4. **FA-04 — Mesmo municipio:** Sub-score distancia = 100 (maximo) por proximidade
5. **FA-05 — Prazo muito curto:** Sub-score prazo <= 20; ponto de atencao "prazo de entrega critico" registrado
6. **FA-06 — Frete inviavel:** Custo de frete supera margem; sub-score frete <= 10 com alerta de inviabilidade
7. **FA-07 — Recalculo apos atualizacao:** Usuario atualiza dados da empresa e recalcula; novos scores refletem alteracoes

## Fluxos de excecao
1. **FE-01 — Edital nao encontrado:** Backend retorna 404; frontend exibe erro de edital invalido
2. **FE-02 — Servico de geolocalizacao indisponivel:** Calculo de distancia usa fallback por UF (menos preciso)
3. **FE-03 — DeepSeek indisponivel:** Inferencia de custos falha; score usa estimativas padrao com flag
4. **FE-04 — Dados de produto incompletos:** Score de frete e prazo recebem penalidade por falta de dados
5. **FE-05 — Token expirado:** Requisicao retorna 401; sistema redireciona para LoginPage
6. **FE-06 — Timeout do calculo:** Calculo excede 30s; backend retorna erro e sugere retry
7. **FE-07 — Divisao por zero:** Margem zero ou prazo zero; sistema aplica valor minimo (1) para evitar erro

## Criterios de validacao
1. Score logistico calculado como media ponderada 30/25/25/20 das sub-dimensoes
2. Cada sub-score individual entre 0 e 100
3. Score logistico compoe corretamente 10% do score geral 6D
4. Edital com orgao proximo recebe score de distancia maior que orgao distante
5. Historico positivo de entregas eleva sub-score de historico
6. Pontos de atencao logisticos sao relevantes e especificos ao edital
7. Score persistido e recuperavel via API
