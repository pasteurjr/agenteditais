# Caso de Uso: UC-019

## Nome
Decisao de participacao

## Objetivo
Permitir ao usuario registrar a decisao final sobre participar ou nao em um edital, com base nos scores de aderencia, validacao das 6 dimensoes e analise de mercado, atualizando o status do edital e encaminhando para as proximas etapas do fluxo comercial.

## Escopo
Modulo Validacao — ValidacaoPage.tsx

## Atores
- Usuario autenticado (Analista Comercial / Gestor)
- Sistema backend (Flask)
- Banco de dados

## Gatilho
Usuario clica no botao "Participar" (go) ou "Nao Participar" (nogo) no painel de decisao da ValidacaoPage, apos revisar scores e analises.

## Pre-condicoes
1. Edital salvo e com score calculado (UC-016 executado)
2. Validacao das 6 dimensoes concluida (UC-017 executado)
3. Analise de mercado disponivel (UC-018 executada, opcional mas recomendada)
4. Usuario tem permissao para registrar decisao (papel de Analista ou Gestor)
5. Dados de scores, pontos positivos e pontos de atencao disponiveis
6. Backend Flask ativo e banco de dados acessivel
7. Edital com status "novo" ou "avaliando" (nao pode decidir sobre edital ja decidido sem reverter)

## Pos-condicoes
1. Status do edital atualizado para "go" (participar) ou "nogo" (nao participar)
2. Decisao registrada no banco com timestamp, usuario responsavel e justificativa
3. Se "go": edital encaminhado para PrecificacaoPage (proximo passo do fluxo)
4. Se "nogo": edital arquivado com motivo registrado
5. Dashboard atualizado com contagem de editais go/nogo
6. Historico de decisoes da empresa atualizado para analytics
7. Notificacao enviada para equipe sobre decisao tomada (se configurado)

## Regras de negocio
1. **RN-DEC-001:** Decisao aceita 3 valores: "go" (participar), "nogo" (nao participar), "avaliando" (manter em analise)
2. **RN-DEC-002:** Decisao requer justificativa textual obrigatoria (minimo 10 caracteres)
3. **RN-DEC-003:** Sistema sugere decisao baseado no score: >=70 sugere GO, 40-69 sugere AVALIAR, <40 sugere NOGO
4. **RN-DEC-004:** Edital com fatal flaw nao pode receber status "go" sem confirmacao explicita do usuario
5. **RN-DEC-005:** Decisao "go" encaminha automaticamente o edital para fila de Precificacao
6. **RN-DEC-006:** Decisao pode ser revertida enquanto edital nao tiver precificacao iniciada
7. **RN-DEC-007:** Historico de decisoes registra: data, usuario, decisao anterior, decisao nova e justificativa

## Componentes de UI envolvidos
1. RadioGroup com opcoes: "Participar (GO)", "Nao Participar (NO-GO)", "Manter em Avaliacao"
2. TextArea para justificativa da decisao (obrigatorio, minimo 10 caracteres)
3. StatusBadge com status atual do edital (novo/avaliando/go/nogo) e cor correspondente
4. Card de resumo — score geral, recomendacao IA, pontos positivos e de atencao
5. Botao "Confirmar Decisao" (ActionButton com icone CheckCircle ou XCircle)
6. Modal de confirmacao para decisoes "go" em editais com fatal flaws
7. Botao "Reverter Decisao" — permite alterar decisao anterior (se edital nao progrediu)

## Endpoints ou servicos envolvidos
1. **POST /api/editais/{id}/decisao** — registra decisao de participacao
2. **crudUpdate('editais', id)** — atualiza status do edital no banco
3. **GET /api/editais/{id}/scores** — scores para exibicao no resumo da decisao
4. **GET /api/editais/{id}/mercado** — dados de mercado para contexto da decisao
5. **crudCreate('decisoes_historico')** — registra historico de decisao para auditoria
6. **crudList('editais', { status: 'go' })** — lista editais aprovados para Dashboard
7. **POST /api/notificacoes** — envia notificacao de decisao para equipe (se configurado)

## Dados manipulados
1. **Decisao:** "go" | "nogo" | "avaliando" — valor da decisao registrada
2. **Justificativa:** string — texto livre com motivo da decisao (minimo 10 caracteres)
3. **Usuario responsavel:** string — ID do usuario que registrou a decisao
4. **Timestamp:** datetime — data e hora do registro da decisao
5. **Score geral:** number — score de aderencia no momento da decisao (para historico)
6. **Recomendacao IA:** string — sugestao do sistema no momento da decisao
7. **Decisao anterior:** string | null — se houve reversao, registra decisao anterior

## Fluxo principal
1. Usuario revisa o painel de validacao do edital com score geral, 6 dimensoes, pontos positivos/atencao e analise de mercado
2. Sistema exibe recomendacao automatica baseada no score (PARTICIPAR/AVALIAR/NAO PARTICIPAR) no StatusBadge
3. Usuario seleciona sua decisao no RadioGroup: "Participar (GO)", "Nao Participar (NO-GO)" ou "Manter em Avaliacao"
4. Usuario digita justificativa da decisao no TextArea (ex: "Portfolio atende 90% dos itens, margem atrativa de 25%")
5. Usuario clica em "Confirmar Decisao" que dispara POST /api/editais/{id}/decisao com status e justificativa
6. Backend persiste decisao no banco, atualiza status do edital e registra no historico de decisoes
7. Frontend atualiza StatusBadge para novo status, exibe toast de confirmacao e encaminha para proxima etapa (Precificacao se GO)

## Fluxos alternativos
1. **FA-01 — Concordar com recomendacao IA:** Usuario seleciona a mesma opcao sugerida pelo sistema; justificativa pode referenciar a analise IA
2. **FA-02 — Discordar da recomendacao:** Usuario decide diferente da sugestao IA; sistema aceita mas registra divergencia
3. **FA-03 — Manter em avaliacao:** Usuario escolhe "Manter em Avaliacao" para retomar analise posteriormente
4. **FA-04 — Reverter decisao:** Usuario clica em "Reverter Decisao" para alterar decisao anterior (go -> nogo ou vice-versa)
5. **FA-05 — Decisao em lote:** Usuario seleciona multiplos editais e aplica decisao em lote (ex: nogo para todos com score < 30)
6. **FA-06 — Aprovacao por gestor:** Decisao "go" em edital de alto valor requer aprovacao adicional do gestor
7. **FA-07 — Decisao via Chat:** Usuario registra decisao pelo FloatingChat informando edital e decisao

## Fluxos de excecao
1. **FE-01 — Justificativa muito curta:** Frontend valida minimo de 10 caracteres e exibe erro inline
2. **FE-02 — Edital sem score:** Sistema impede decisao e exibe alerta "Execute a validacao antes de decidir"
3. **FE-03 — GO com fatal flaw:** Modal de confirmacao exibe warning com lista de fatal flaws e pede confirmacao explicita
4. **FE-04 — Edital ja decidido:** Se edital ja tem decisao e progrediu para Precificacao, reversao bloqueada com mensagem
5. **FE-05 — Erro de persistencia:** Backend retorna erro 500; frontend exibe toast de erro e mantem formulario preenchido
6. **FE-06 — Sessao expirada:** Token JWT invalido; sistema redireciona para login e perde formulario
7. **FE-07 — Concorrencia de decisao:** Dois usuarios tentam decidir sobre mesmo edital simultaneamente; sistema aplica ultima escrita

2. Justificativa e obrigatoria com minimo de 10 caracteres
3. Decisao "go" atualiza status do edital para "go" e encaminha para Precificacao
4. Decisao "nogo" atualiza status para "nogo" e arquiva edital
5. Historico de decisao e registrado com timestamp e usuario
6. Fatal flaws exibem modal de confirmacao antes de permitir GO
7. Reversao de decisao funciona quando edital nao progrediu no fluxo
