# Caso de Uso: UC-019

## Nome
Listar editais salvos, selecionar edital e usar IA na validacao

## Objetivo
Cobrir duas funcionalidades da ValidacaoPage: (1) listagem e selecao de editais salvos para analise — ponto de entrada da pagina de validacao; (2) uso da IA na validacao via aba "IA" — geracao de resumo, perguntas livres e acoes rapidas sobre o edital selecionado.

## Escopo
Modulo Validacao — ValidacaoPage.tsx

## Atores
- Usuario autenticado (Analista Comercial / Gestor)
- Sistema backend (Flask)
- Banco de dados

## Gatilho
- Listagem: usuario acessa a ValidacaoPage
- Selecao: usuario clica numa linha da tabela "Meus Editais"
- IA: usuario abre a aba "IA" e clica em "Gerar Resumo", "Perguntar", "Requisitos Tecnicos" ou "Classificar Edital"

## Pre-condicoes
1. Editais ja salvos na captacao (UC-015 executado)
2. Endpoint GET /api/editais/salvos disponivel
3. Backend Flask ativo e banco de dados acessivel
4. Para a aba "IA": edital selecionado e servico de chat/IA disponivel

## Pos-condicoes
1. Usuario consegue selecionar um edital salvo para analisar
2. Painel por abas da validacao passa a refletir o edital escolhido (itens, lotes, historico e documentacao sao carregados)
3. Na aba "IA": usuario consegue resumir, perguntar e obter respostas especificas sobre o edital
4. Respostas ficam refletidas na propria aba "IA"

## Regras de negocio
1. **RN-VAL-001:** Listagem carrega editais via GET /api/editais/salvos?com_score=true&com_estrategia=true
2. **RN-VAL-002:** A selecao de um edital limpa todos os estados auxiliares antes de montar o painel de analise
3. **RN-VAL-003:** Apos selecao, o sistema tenta automaticamente carregar itens, lotes, historico e documentacao necessaria
4. **RN-VAL-004:** O filtro de status da tabela permite filtrar por: novo, avaliando, go, nogo e outros
5. **RN-VAL-005:** Na aba "IA", o chat usa sessao de pagina para contextualizar respostas ao edital selecionado
6. **RN-VAL-006:** As acoes rapidas "Requisitos Tecnicos" e "Classificar Edital" enviam prompts pre-definidos para /api/chat

## Componentes de UI envolvidos

Listagem e selecao (UC-CV07):
1. Tabela "Meus Editais" — lista de editais salvos
2. Filtro por status na tabela
3. Campo de pesquisa por texto
4. Clique na linha do edital para selecionar
5. Botao "Tentar novamente" em caso de falha de carga
6. Botao "Ver Edital" — abre PdfViewer para pdfUrl ou pdfPath
7. Select de status do edital no cabecalho da analise

Aba "IA" (UC-CV13):
8. Botao "Gerar Resumo" / "Regerar Resumo"
9. Campo de texto para pergunta + Botao "Perguntar"
10. Botao "Requisitos Tecnicos" (acao rapida)
11. Botao "Classificar Edital" (acao rapida)
12. Area de exibicao das respostas da IA

## Endpoints ou servicos envolvidos

Listagem e selecao:
1. **GET /api/editais/salvos?com_score=true&com_estrategia=true** — carrega editais salvos com dados de score e estrategia
2. **crudList('editais')** — listagem base via CRUD generico

Aba "IA":
3. **POST /api/chat** — todas as interacoes da aba IA (resumo, perguntas, acoes rapidas)
4. Sessao de pagina — contexto da conversa e mantido por edital selecionado

Nota sobre decisao (DIVERGENCIA):
5. **crudCreate("validacao_decisoes", ...)** — chamada existente no frontend para salvar decisao GO/NO-GO (tabela NAO existe no schema MySQL `editais`)
6. **crudUpdate("validacao_decisoes", ...)** — chamada existente no frontend para atualizar decisao (tabela NAO existe no schema MySQL `editais`)

## Dados manipulados
1. **Editais salvos:** lista completa via GET /api/editais/salvos com score e estrategia
2. **Filtro de status:** string — status selecionado na tabela (novo, avaliando, go, nogo)
3. **Edital selecionado:** objeto EditalSalvo completo com todos os campos
4. **Sessao de pagina IA:** contexto da conversa mantido por edital selecionado
5. **Perguntas e respostas:** string[] — historico da conversa na aba "IA"
6. **Resumo do edital:** string — gerado pela IA e exibido na aba "IA"

## Fluxo principal — Listagem e Selecao (UC-CV07)
1. Usuario acessa ValidacaoPage
2. Sistema carrega editais via GET /api/editais/salvos?com_score=true&com_estrategia=true
3. Usuario filtra por status ou pesquisa por texto na tabela "Meus Editais"
4. Usuario clica numa linha da tabela
5. Sistema limpa estados auxiliares e monta o painel de analise do edital selecionado
6. A partir da selecao, a tela tambem tenta carregar itens, lotes, historico e documentacao necessaria
7. Usuario pode clicar em "Ver Edital" para abrir PdfViewer (pdfUrl ou pdfPath)

## Fluxo alternativo — Aba IA (UC-CV13)
1. Usuario abre a aba "IA"
2. Usuario gera ou regera um resumo do edital via POST /api/chat
3. Usuario digita uma pergunta e clica em "Perguntar"
4. Usuario tambem pode usar as acoes rapidas "Requisitos Tecnicos" e "Classificar Edital"
5. Sistema usa POST /api/chat com a sessao de pagina para devolver respostas contextualizadas ao edital selecionado

## Fluxos alternativos
1. **FA-01 — Concordar com recomendacao IA:** Usuario seleciona a mesma opcao sugerida pelo sistema; justificativa pode referenciar a analise IA
2. **FA-02 — Discordar da recomendacao:** Usuario decide diferente da sugestao IA; sistema aceita mas registra divergencia
3. **FA-03 — Manter em avaliacao:** Usuario escolhe "Manter em Avaliacao" para retomar analise posteriormente
4. **FA-04 — Reverter decisao:** Usuario clica em "Reverter Decisao" para alterar decisao anterior (go -> nogo ou vice-versa)
5. **FA-05 — Decisao em lote:** Usuario seleciona multiplos editais e aplica decisao em lote (ex: nogo para todos com score < 30)
6. **FA-06 — Aprovacao por gestor:** Decisao "go" em edital de alto valor requer aprovacao adicional do gestor
7. **FA-07 — Decisao via Chat:** Usuario registra decisao pelo FloatingChat informando edital e decisao

## Fluxos de excecao
1. **FE-01 — Justificativa muito curta:** Frontend valida minimo de 10 caracteres e exibe erro inline
2. **FE-02 — Edital sem score:** Sistema impede decisao e exibe alerta "Execute a validacao antes de decidir"
3. **FE-03 — GO com fatal flaw:** Modal de confirmacao exibe warning com lista de fatal flaws e pede confirmacao explicita
4. **FE-04 — Edital ja decidido:** Se edital ja tem decisao e progrediu para Precificacao, reversao bloqueada com mensagem
5. **FE-05 — Erro de persistencia:** Backend retorna erro 500; frontend exibe toast de erro e mantem formulario preenchido
6. **FE-06 — Sessao expirada:** Token JWT invalido; sistema redireciona para login e perde formulario
7. **FE-07 — Concorrencia de decisao:** Dois usuarios tentam decidir sobre mesmo edital simultaneamente; sistema aplica ultima escrita

## Criterios de validacao
1. RadioGroup permite selecao de apenas uma opcao (GO/NOGO/AVALIAR)
2. Justificativa e obrigatoria com minimo de 10 caracteres
3. Decisao "go" atualiza status do edital para "go" e encaminha para Precificacao
4. Decisao "nogo" atualiza status para "nogo" e arquiva edital
5. Historico de decisao e registrado com timestamp e usuario
6. Fatal flaws exibem modal de confirmacao antes de permitir GO
7. Reversao de decisao funciona quando edital nao progrediu no fluxo

## Divergencias encontradas
- O UC original descrevia "Decisao de participacao" com RadioGroup GO/NOGO, persistencia e historico — esse fluxo na verdade pertence ao UC-016 (aba Aderencia da ValidacaoPage)
- **CRITICO (herdado do UC-016):** O frontend possui chamadas a `crudCreate("validacao_decisoes", ...)` e `crudUpdate("validacao_decisoes", ...)` para persistir a decisao, mas a tabela `validacao_decisoes` NAO existe no schema MySQL `editais` consultado em 30/03/2026. Essa divergencia afeta tanto o UC-016 quanto qualquer logica de decisao referenciada neste UC
- O conteudo real deste UC e a entrada na ValidacaoPage (listagem/selecao) e a aba "IA" — ambos sem divergencias criticas de implementacao
