# Caso de Uso: UC-043

## Nome
Configurar alertas de vencimento

## Objetivo
Permitir ao usuario configurar regras de alerta multi-tier (30d/15d/7d/1d) para vencimentos de contratos, ARPs e entregas, com consolidacao visual por urgencia (vermelho/laranja/amarelo/verde) e canais de notificacao (email/push).

## Escopo
Modulo Follow-up — FollowupPage.tsx > Aba Alertas

## Atores
- Usuario autenticado (Gestor de Contratos)
- Sistema backend (Flask)
- Scheduler de alertas (verificacao periodica)
- Servico de notificacoes (email/push)

## Gatilho
Usuario acessa a aba "Alertas" na FollowupPage para visualizar vencimentos consolidados e configurar regras de alerta por tipo de entidade.

## Pre-condicoes
1. Usuario esta autenticado no sistema
2. Backend Flask ativo com endpoints de alertas de vencimento
3. Ao menos um contrato, ARP ou entrega cadastrados com data de vencimento
4. Tabela de regras de alerta criada no banco de dados
5. Endpoint /api/alertas-vencimento/regras acessivel
6. Endpoint /api/alertas-vencimento/consolidado acessivel
7. Scheduler configurado para verificacao periodica de vencimentos

## Pos-condicoes
1. Regras de alerta persistidas com tiers selecionados (30d/15d/7d/1d)
2. Canais de notificacao configurados (email e/ou push) por tipo de entidade
3. Vencimentos consolidados exibidos com semaforo de urgencia
4. Resumo visual com totais por cor (vermelho/laranja/amarelo/verde)
5. Alertas disparam automaticamente nos dias configurados
6. Regras podem ser ativadas/desativadas individualmente
7. Dashboard de vencimentos acessivel tanto na FollowupPage quanto na ContratadoRealizadoPage

## Regras de negocio
1. **RN-ALE-001:** 4 tiers de alerta: 30 dias, 15 dias, 7 dias, 1 dia antes do vencimento
2. **RN-ALE-002:** Urgencia por cor: vermelho (<=7d), laranja (8-15d), amarelo (16-30d), verde (>30d)
3. **RN-ALE-003:** Tipos de entidade monitorados: contrato, arp, entrega
4. **RN-ALE-004:** Canais de notificacao: canal_email e canal_push, configurados por regra
5. **RN-ALE-005:** Regra pode ser ativada/desativada via toggle booleano "ativo"
6. **RN-ALE-006:** Consolidado agrupa vencimentos de todas as entidades em lista unica ordenada por dias_restantes
7. **RN-ALE-007:** Resumo contabiliza total e quantidade por cada faixa de urgencia

## Componentes de UI envolvidos
1. TabPanel aba "Alertas" com icone Bell na FollowupPage
2. StatCards de resumo: total, vermelho, laranja, amarelo, verde (vencResumo)
3. DataTable de vencimentos com colunas: tipo_entidade, nome, data_vencimento, dias_restantes, urgencia
4. Badges de urgencia coloridos (vermelho/laranja/amarelo/verde)
5. Tabela de regras de alerta com toggles para cada tier (30d/15d/7d/1d)
6. Checkboxes para canais de notificacao (email/push)
7. Toggle de ativo/desativado por regra de alerta

## Endpoints ou servicos envolvidos
1. **GET /api/alertas-vencimento/regras** — lista regras de alerta configuradas
2. **GET /api/alertas-vencimento/consolidado** — vencimentos consolidados com resumo
3. **POST /api/alertas-vencimento/regras** — cria nova regra de alerta
4. **PUT /api/alertas-vencimento/regras/{id}** — atualiza regra existente
5. **DELETE /api/alertas-vencimento/regras/{id}** — remove regra de alerta
6. **Scheduler interno** — verifica diariamente vencimentos e dispara alertas
7. **Servico de notificacoes** — envia email/push conforme regras ativas

## Dados manipulados
1. **Regra de alerta:** id, tipo_entidade, dias_30 (bool), dias_15, dias_7, dias_1, canal_email, canal_push, ativo
2. **Vencimento:** tipo_entidade, nome, data_vencimento, dias_restantes, urgencia (cor)
3. **Resumo:** total, vermelho (count), laranja (count), amarelo (count), verde (count)
4. **Tipo entidade:** "contrato" | "arp" | "entrega"
5. **Urgencia:** "vermelho" (<=7d) | "laranja" (8-15d) | "amarelo" (16-30d) | "verde" (>30d)
6. **Canal:** canal_email (boolean), canal_push (boolean)
7. **Estado ativo:** boolean — permite desativar regra sem excluir

## Fluxo principal
1. Usuario acessa FollowupPage e clica na aba "Alertas" que carrega fetchAlertas() em paralelo (regras + consolidado)
2. Sistema exibe StatCards com resumo de vencimentos: total, vermelhos (criticos), laranjas (urgentes), amarelos (atencao), verdes (ok)
3. DataTable mostra lista de vencimentos consolidados ordenados por dias_restantes, com badges de urgencia coloridos
4. Usuario visualiza tabela de regras de alerta por tipo_entidade com checkboxes para tiers (30d/15d/7d/1d)
5. Usuario ativa/desativa tiers e canais (email/push) conforme necessidade de cada tipo de entidade
6. Sistema persiste alteracoes via PUT /api/alertas-vencimento/regras/{id} ao confirmar mudancas
7. Scheduler processa regras ativas diariamente e dispara notificacoes nos dias configurados

## Fluxos alternativos
1. **FA-01 — Sem vencimentos proximos:** Consolidado retorna lista vazia; StatCards mostram zeros; mensagem informativa exibida
2. **FA-02 — Criar nova regra:** Usuario clica em "Nova Regra" para tipo_entidade nao configurado; modal de criacao abre
3. **FA-03 — Desativar todos os alertas:** Usuario desativa toggle "ativo" em todas as regras; sistema para de disparar notificacoes
4. **FA-04 — Apenas email:** Usuario marca canal_email e desmarca canal_push; alertas so vao por email
5. **FA-05 — Vencimento ja expirado:** Item com dias_restantes negativo aparece como "vencido ha X dias" com urgencia vermelha
6. **FA-06 — Filtrar por tipo:** Usuario filtra vencimentos por tipo_entidade (contrato/arp/entrega)
7. **FA-07 — Acesso via ContratadoRealizadoPage:** Mesmos dados de vencimento acessiveis na ContratadoRealizadoPage (UC-056)

## Fluxos de excecao
1. **FE-01 — Erro ao carregar regras:** GET /api/alertas-vencimento/regras falha; try/catch exibe erro no console
2. **FE-02 — Erro ao carregar consolidado:** GET /api/alertas-vencimento/consolidado falha; StatCards ficam zerados
3. **FE-03 — Token expirado:** Requisicao retorna 401; sistema redireciona para LoginPage
4. **FE-04 — Scheduler indisponivel:** Alertas nao sao disparados automaticamente; dados consolidados ainda visiveis manualmente
5. **FE-05 — Email service falha:** Alerta marcado como "falha_envio" no log; retry na proxima execucao do scheduler
6. **FE-06 — Regra duplicada:** Tentativa de criar regra para tipo_entidade ja configurado; backend retorna erro de conflito
7. **FE-07 — Dados corrompidos de vencimento:** data_vencimento nula ou invalida; sistema ignora item e loga warning

11. FU02-F12 atualiza com novo alerta; FU02-F13/F14 permitem editar/desativar
