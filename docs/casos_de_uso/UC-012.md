# Caso de Uso: UC-012

## Nome
Dashboard Geral

## Objetivo
Fornecer uma visao consolidada e em tempo real do estado de todos os editais e processos licitatorios, incluindo KPIs, funil de vendas, editais urgentes, status do scheduler de monitoramento e notificacoes.

## Escopo
Modulo Dashboard — pagina inicial do sistema

## Atores
- Usuario (operador do sistema)
- Sistema (scheduler de monitoramento automatico)

## Gatilho
O usuario acessa o sistema e visualiza o Dashboard (pagina inicial) ou navega para o Dashboard a partir de qualquer outra pagina.

## Pre-condicoes
1. Usuario autenticado no sistema
2. Backend ativo na porta 5007
3. Endpoint /api/dashboard/stats disponivel
4. Ao menos alguns editais cadastrados para dados significativos

## Pos-condicoes
1. Dashboard exibido com dados atualizados
2. KPIs calculados e apresentados
3. Funil de vendas renderizado com quantidades por etapa
4. Editais urgentes listados com prazos
5. Status do scheduler visivel

## Regras de negocio
- RN-UC012-001: KPIs calculados a partir de editais_por_status e propostas_por_status retornados pelo backend
- RN-UC012-002: Funil de vendas mapeia etapas: Captacao -> Validacao -> Precificacao -> Proposta -> Submissao -> Ganhos
- RN-UC012-003: Editais urgentes sao os com proximos prazos de abertura (proximos_prazos do backend)
- RN-UC012-004: Taxa de sucesso = (ganhos / total editais) * 100
- RN-UC012-005: Valor total contratado exibido em formato abreviado (ex: "R$ 150k")
- RN-UC012-006: Status bar exibe contadores rapidos: novos, em analise, validados, propostas enviadas, lance hoje
- RN-UC012-007: Scheduler de monitoramento exibe: status ativo/inativo, monitoramentos ativos/inativos, notificacoes nao lidas
- RN-UC012-008: Notificacoes sao carregadas de /api/notificacoes/nao-lidas
- RN-UC012-009: Busca rapida permite navegar diretamente para outros modulos
- RN-UC012-010: Dashboard atualiza dados automaticamente ao ser carregado

## Componentes de UI envolvidos
- Dashboard.tsx (componente principal)
- Card de KPIs: emAnalise, propostas, ganhos, taxaSucesso, valorTotal
- Funil de vendas visual: 6 etapas com contadores (captacao, validacao, precificacao, proposta, submissao, ganhos)
- Lista de editais urgentes: EditalUrgente[] com numero, orgao, prazo, valor
- Status bar: novos, emAnalise, validados, propostasEnviadas, lanceHoje
- Card de eventos/calendario: proximos prazos com dia, mes, titulo, subtitulo
- Painel de scheduler: schedulerStatus com monitoramentos ativos, frequencia, ultima execucao, proximo check
- Icone Bell com badge de notificacoes nao lidas (notificacoes)
- TextInput de busca rapida (buscaRapida)
- Icones: AlertTriangle, Clock, FileText, TrendingUp, CheckCircle, DollarSign, Target, Calendar, Bell, Search, Sparkles, Brain
- Loader2 durante carregamento (loading)
- AlertCircle para estado de erro (error)

## Endpoints ou servicos envolvidos
- GET /api/dashboard/stats — estatisticas consolidadas (fetchDashboardStats)
  - Retorna: editais_por_status, propostas_por_status, proximos_prazos, total_propostas, taxa_sucesso, valor_total_contratado
- GET /api/scheduler/status — status do scheduler de monitoramento (loadSchedulerStatus)
  - Retorna: scheduler_ativo, scheduler_disponivel, monitoramentos_ativos, monitoramentos_inativos, notificacoes_nao_lidas, monitoramentos[]
- GET /api/notificacoes/nao-lidas — lista notificacoes nao lidas
- Autenticacao: Token Bearer via localStorage (editais_ia_access_token) ou getDashboardTokenFn

## Dados manipulados
- DashboardStats: urgentes[], funil (FunilData), kpis (KpisData), statusBar (StatusBarData), insights[], eventos[], aprendizados[]
- KpisData: emAnalise (number), propostas (number), ganhos (number), taxaSucesso (string ex: "45%"), valorTotal (string ex: "R$ 150k")
- FunilData: captacao, validacao, precificacao, proposta, submissao, ganhos (todos number)
- StatusBarData: novos, emAnalise, validados, propostasEnviadas, lanceHoje (todos number)
- EditalUrgente: id, numero, orgao, prazo, valor
- SchedulerStatus: scheduler_ativo, scheduler_disponivel, monitoramentos_ativos, monitoramentos_inativos, notificacoes_nao_lidas, monitoramentos[]
- NotificacaoItem: id, tipo, titulo, mensagem, created_at

## Fluxo principal

### Passo 1: Acessar Dashboard
**Acao do ator:** O usuario acessa o sistema ou clica no menu "Dashboard".
**Elemento UI:** Pagina inicial do sistema com componente Dashboard.
**Resposta visual:** Loader2 exibido durante carregamento (loading === true). Apos carregamento, Dashboard completo renderizado com todos os cards e secoes.
**Efeito funcional:** fetchDashboardStats() chamado no useEffect. loadSchedulerStatus() chamado em paralelo.
**Requisicao de rede:** GET /api/dashboard/stats (com Authorization Bearer token). GET /api/scheduler/status. GET /api/notificacoes/nao-lidas.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot do Dashboard completo com todos os cards carregados.

### Passo 2: Visualizar KPIs
**Acao do ator:** O usuario observa os cards de KPIs na parte superior do Dashboard.
**Elemento UI:** Cards com icones: FileText (em analise), Target (propostas), CheckCircle (ganhos), TrendingUp (taxa de sucesso), DollarSign (valor total).
**Resposta visual:** Cada card exibe: titulo, valor numerico grande, icone contextual. Valores calculados a partir de editais_por_status e propostas_por_status.
**Efeito funcional:** KPIs derivados: emAnalise = eps.analisando, propostas = total_propostas, ganhos = eps.ganho + eps.vencedor, taxaSucesso = Math.round(taxa_sucesso * 100) + "%", valorTotal = "R$ " + (valor_total_contratado / 1000).toFixed(0) + "k".
**Requisicao de rede:** Nenhuma (dados ja carregados no Passo 1).
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot dos cards de KPIs com valores.

### Passo 3: Visualizar funil de vendas
**Acao do ator:** O usuario observa o funil de vendas visual.
**Elemento UI:** Componente visual de funil com 6 etapas sequenciais: Captacao -> Validacao -> Precificacao -> Proposta -> Submissao -> Ganhos. Cada etapa com contador numerico.
**Resposta visual:** Funil renderizado com barras ou segmentos decrescentes. Numeros por etapa: captacao (eps.novo), validacao (eps.analisando), precificacao (eps.precificando), proposta (rascunho+revisao+aprovada), submissao (enviada), ganhos.
**Efeito funcional:** FunilData mapeado a partir de editais_por_status e propostas_por_status.
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot do funil de vendas com numeros por etapa.

### Passo 4: Verificar editais urgentes
**Acao do ator:** O usuario observa a secao de editais urgentes (proximos prazos).
**Elemento UI:** Lista/tabela de editais com prazo proximo. Colunas: numero do edital, orgao, prazo (dias restantes), valor. Icone AlertTriangle para urgencia.
**Resposta visual:** Editais listados com indicacao de prazo (ex: "3 dias", "7 dias"). Editais com prazo curto destacados com cores de alerta.
**Efeito funcional:** proximos_prazos do backend mapeados para EditalUrgente[].
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot da lista de editais urgentes com prazos.

### Passo 5: Verificar status bar
**Acao do ator:** O usuario observa a barra de status rapido.
**Elemento UI:** Barra horizontal com badges/contadores: Novos, Em Analise, Validados, Propostas Enviadas, Lance Hoje.
**Resposta visual:** Cada badge exibe contagem com icone e cor correspondente.
**Efeito funcional:** StatusBarData derivado de editais_por_status e propostas_por_status.
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot da barra de status com contadores.

### Passo 6: Verificar status do scheduler
**Acao do ator:** O usuario observa o painel do scheduler de monitoramento.
**Elemento UI:** Card de scheduler com: status ativo/inativo, quantidade de monitoramentos ativos, frequencia, ultima execucao, proximo check. Lista de monitoramentos com termo, NCM, UFs, editais encontrados.
**Resposta visual:** Badge verde (ativo) ou cinza (inativo). Contadores de monitoramentos. Lista com detalhes de cada monitoramento.
**Efeito funcional:** schedulerStatus carregado de /api/scheduler/status.
**Requisicao de rede:** Ja carregado no Passo 1 (loadSchedulerStatus).
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot do painel de scheduler com status e monitoramentos.

### Passo 7: Verificar notificacoes
**Acao do ator:** O usuario clica no icone de notificacoes (Bell) para ver notificacoes nao lidas.
**Elemento UI:** Icone Bell com badge numerico (contagem de nao lidas). Panel/dropdown de notificacoes (showNotifs).
**Resposta visual:** Badge exibe numero de notificacoes nao lidas. Ao clicar, lista de notificacoes aparece com titulo, mensagem e data.
**Efeito funcional:** showNotifs toggled. notificacoes exibidas.
**Requisicao de rede:** GET /api/notificacoes/nao-lidas (ja carregado ou recarregado ao clicar).
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot do painel de notificacoes aberto.

### Passo 8: Usar busca rapida
**Acao do ator:** O usuario digita um termo na busca rapida e navega para o modulo correspondente.
**Elemento UI:** TextInput de busca rapida (buscaRapida) com icone Search.
**Resposta visual:** Campo de busca aceita texto. Ao pressionar Enter ou clicar, navega para modulo correspondente.
**Efeito funcional:** onNavigate(page) chamado para redirecionar ao modulo.
**Requisicao de rede:** Nenhuma (navegacao local).
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot da busca rapida com termo digitado.

## Fluxos alternativos

### FA-001: Dashboard vazio (sem dados)
**Condicao:** Nao ha editais cadastrados no sistema.
**Variacao:** KPIs exibem zeros. Funil vazio. Lista de urgentes vazia. Status bar com todos os contadores zerados. Sistema exibe estado vazio com sugestoes (ex: "Comece cadastrando seus produtos" ou "Busque editais na Captacao").

### FA-002: Scheduler inativo
**Condicao:** Scheduler de monitoramento nao esta ativo ou nao disponivel.
**Variacao:** Card de scheduler exibe badge "Inativo" em cinza. Contagem de monitoramentos zerada. Opcao de ativar pode ser exibida.

### FA-003: Navegacao rapida pelo Dashboard
**Condicao:** Usuario clica em um card de KPI ou etapa do funil para navegar.
**Variacao:** onNavigate chamado com pagina correspondente (ex: clicar em "Em Analise" navega para ValidacaoPage).

## Fluxos de excecao

### FE-001: Erro ao carregar estatisticas
**Condicao:** GET /api/dashboard/stats retorna erro.
**Resposta:** Estado error recebe mensagem. Dashboard exibe AlertCircle com mensagem de erro. Botao RefreshCw para tentar novamente.

### FE-002: Nao autenticado (401)
**Condicao:** Token expirado ou invalido.
**Resposta:** fetchDashboardStats lanca erro "Nao autenticado". Sistema redireciona para login.

### FE-003: Scheduler indisponivel
**Condicao:** GET /api/scheduler/status retorna erro.
**Resposta:** Painel de scheduler nao exibido ou exibe estado indeterminado. Nao bloqueia o carregamento do restante do Dashboard.
