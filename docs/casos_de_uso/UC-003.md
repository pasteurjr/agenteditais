# Caso de Uso: UC-003

## Nome
Gestao de Documentos da Empresa

## Objetivo
Permitir que o usuario faca upload, visualize, organize e gerencie os documentos da empresa (contrato social, alvaras, certidoes, entre outros), categorizados por tipo, com controle de validade e status.

## Escopo
Modulo Empresa — EmpresaPage.tsx > aba Documentos / Backend CRUD empresa-documentos

## Atores
- Ator primario: Usuario autenticado (gestor documental da empresa)

## Gatilho
O usuario acessa o menu "Empresa" e clica na aba "Documentos".

## Pre-condicoes
1. O usuario esta autenticado com JWT valido
2. Existe pelo menos uma empresa cadastrada vinculada ao usuario
3. O sistema (frontend e backend) esta operacional

## Pos-condicoes
1. Os documentos da empresa estao persistidos na tabela "empresa_documentos"
2. Cada documento possui tipo, nome, validade e status (ok, vence, falta)
3. Os documentos podem ser visualizados e baixados

## Regras de negocio
- RN-DOC-001: Documentos sao organizados por categorias configuraveis (via tabela categorias_documento)
- RN-DOC-002: Cada documento tem status: "ok" (valido), "vence" (proximo do vencimento) ou "falta" (nao enviado/vencido)
- RN-DOC-003: Documentos necessarios sao definidos na tabela documentos_necessarios, com flag de obrigatoriedade
- RN-DOC-004: Upload aceita formatos comuns (PDF, DOC, DOCX, imagens)
- RN-DOC-005: Documentos sao scoped por empresa do usuario

## Componentes de UI envolvidos
- EmpresaPage > aba "Documentos"
- DataTable com colunas: nome, tipo, validade, status, acoes
- StatusBadge para indicar status do documento (ok/vence/falta)
- ActionButton com icone Upload para envio de documento
- ActionButton com icone Eye para visualizacao
- ActionButton com icone Download para download
- ActionButton com icone Trash2 para exclusao
- Modal de upload com campos: arquivo, tipo/categoria, validade
- SelectInput para selecao de tipo/categoria do documento

## Endpoints ou servicos envolvidos
- GET /api/crud/empresa-documentos — lista documentos da empresa
- POST /api/crud/empresa-documentos — cria registro de documento (com upload)
- PUT /api/crud/empresa-documentos/{id} — atualiza metadados do documento
- DELETE /api/crud/empresa-documentos/{id} — remove documento
- GET /api/crud/categorias-documento — lista categorias disponiveis
- GET /api/crud/documentos-necessarios — lista documentos obrigatorios/opcionais

## Dados manipulados
- Entrada: arquivo (File), nome (string), tipo/categoria (string), validade (date | null)
- Saida: objeto Documento com id, nome, tipo, validade, status, path do arquivo
- Armazenamento: tabela "empresa_documentos" no MySQL + arquivo fisico no servidor

## Fluxo principal

### Passo 1: Acesso a aba Documentos
- **Acao:** O ator clica na aba "Documentos" dentro da EmpresaPage
- **Elemento UI:** Tab "Documentos" na barra de abas da EmpresaPage
- **Resposta visual:** O sistema renderiza a lista de documentos da empresa em DataTable; se nao ha documentos, exibe estado vazio; exibe tambem a lista de documentos necessarios com seus status
- **Efeito funcional:** GET /api/crud/empresa-documentos e invocado para carregar documentos existentes
- **Requisicao:** GET /api/crud/empresa-documentos com filtro por empresa_id
- **Persistencia:** Nenhuma (apenas leitura)
- **Evidencia:** Screenshot da aba Documentos (runtime/screenshots/UC-003/01_aba_documentos.png)

### Passo 2: Iniciar upload de documento
- **Acao:** O ator clica no botao "Upload" ou "Novo Documento"
- **Elemento UI:** ActionButton com icone Upload ou Plus
- **Resposta visual:** O sistema abre modal com campos para selecionar arquivo, informar tipo/categoria e validade
- **Efeito funcional:** Modal de upload e exibido
- **Requisicao:** GET /api/crud/categorias-documento para popular o select de categorias
- **Persistencia:** Nenhuma
- **Evidencia:** Screenshot do modal de upload (runtime/screenshots/UC-003/02_modal_upload.png)

### Passo 3: Selecionar arquivo
- **Acao:** O ator clica no campo de upload e seleciona um arquivo do computador
- **Elemento UI:** Input file com area de drag-and-drop
- **Resposta visual:** O nome do arquivo selecionado e exibido no campo
- **Efeito funcional:** O arquivo e carregado no estado local do componente
- **Requisicao:** Nenhuma
- **Persistencia:** Nenhuma
- **Evidencia:** Screenshot com arquivo selecionado (runtime/screenshots/UC-003/03_arquivo_selecionado.png)

### Passo 4: Selecionar tipo do documento
- **Acao:** O ator seleciona o tipo/categoria do documento no dropdown
- **Elemento UI:** SelectInput com opcoes carregadas de categorias_documento
- **Resposta visual:** O dropdown exibe a opcao selecionada
- **Efeito funcional:** Estado local atualizado com o tipo selecionado
- **Requisicao:** Nenhuma
- **Persistencia:** Nenhuma
- **Evidencia:** Screenshot com tipo selecionado (runtime/screenshots/UC-003/04_tipo_selecionado.png)

### Passo 5: Informar validade (opcional)
- **Acao:** O ator informa a data de validade do documento, se aplicavel
- **Elemento UI:** Input date para validade
- **Resposta visual:** Data exibida no campo
- **Efeito funcional:** Estado local atualizado com a data de validade
- **Requisicao:** Nenhuma
- **Persistencia:** Nenhuma
- **Evidencia:** Screenshot com validade preenchida (runtime/screenshots/UC-003/05_validade.png)

### Passo 6: Salvar documento
- **Acao:** O ator clica no botao "Salvar" do modal
- **Elemento UI:** ActionButton "Salvar" no modal
- **Resposta visual:** O modal fecha; a listagem de documentos e atualizada com o novo documento; o StatusBadge correspondente reflete o status "ok"
- **Efeito funcional:** O arquivo e enviado ao backend junto com os metadados
- **Requisicao:** POST /api/crud/empresa-documentos com multipart/form-data (arquivo + metadados)
- **Persistencia:** Registro criado na tabela "empresa_documentos"; arquivo salvo no filesystem do servidor
- **Evidencia:** Screenshot da listagem atualizada (runtime/screenshots/UC-003/06_documento_salvo.png)

### Passo 7: Verificacao na listagem
- **Acao:** O ator observa o documento na tabela
- **Elemento UI:** DataTable com nova linha contendo nome, tipo, validade, status e acoes
- **Resposta visual:** O documento aparece com StatusBadge "ok" (verde), botoes de visualizar, baixar e excluir
- **Efeito funcional:** Listagem reflete o estado atual do banco
- **Requisicao:** Nenhuma (ja atualizado pelo passo anterior)
- **Persistencia:** Nenhuma
- **Evidencia:** Screenshot da listagem com documento (runtime/screenshots/UC-003/07_listagem_com_documento.png)

## Fluxos alternativos

### FA-001: Visualizar documento existente
1. O ator clica no botao Eye de um documento na listagem
2. O sistema abre o documento em nova aba ou modal de preview
3. O ator visualiza o conteudo do documento

### FA-002: Baixar documento
1. O ator clica no botao Download de um documento
2. O sistema inicia o download do arquivo
3. O arquivo e salvo no computador do ator

### FA-003: Excluir documento
1. O ator clica no botao Trash2 de um documento
2. O sistema exibe confirmacao de exclusao
3. O ator confirma
4. DELETE /api/crud/empresa-documentos/{id} e invocado
5. Documento e removido da listagem e do servidor

## Fluxos de excecao

### FE-001: Arquivo invalido ou muito grande
1. O ator seleciona um arquivo com formato nao suportado ou que excede o tamanho maximo
2. O sistema exibe mensagem de erro informando a restricao
3. O upload nao e processado

### FE-002: Erro de rede durante upload
1. O upload e iniciado mas a conexao falha
2. O sistema exibe mensagem de erro
3. O documento nao e salvo; o ator pode tentar novamente

### FE-003: Tipo de documento nao selecionado
1. O ator tenta salvar sem selecionar o tipo/categoria
2. O sistema exibe validacao indicando campo obrigatorio
3. O upload nao e processado
