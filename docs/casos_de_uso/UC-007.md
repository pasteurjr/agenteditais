# Caso de Uso: UC-007

## Nome
Upload de Manual com Extracao IA

## Objetivo
Permitir que o usuario faca upload de um documento tecnico (manual, IFU, folder, NF, plano de contas) ou informe URL de fabricante, para que a IA extraia automaticamente as especificacoes do produto e o cadastre no portfolio.

## Escopo
Modulo Portfolio — PortfolioPage.tsx > aba `Cadastro por IA` / Backend chat/IA (sessao `cadastro-produto`)

## Atores
- Usuario (operador do sistema)
- IA (DeepSeek — extracao automatica de especificacoes)

## Gatilho
O usuario acessa a aba "Cadastro por IA" na PortfolioPage e seleciona um tipo de documento para upload.

## Pre-condicoes
1. Usuario autenticado no sistema
2. Hierarquia de areas, classes e subclasses ja cadastrada (ao menos parcialmente)
3. Backend ativo na porta 5007 com endpoint de chat/IA disponivel
4. Chave de API do DeepSeek configurada no .env

## Pos-condicoes
1. Produto criado no banco de dados com campos extraidos pela IA
2. Especificacoes tecnicas associadas ao produto
3. Score de completude calculado para o novo produto
4. Produto visivel na aba "Produtos" da PortfolioPage

## Regras de negocio
- RN-PORT-003: Upload de NF e Plano de Contas cria multiplos produtos de uma vez
- RN-PORT-004: Cada produto tem NCM, fabricante, modelo como campos obrigatorios
- RN-UC007-001: Seis tipos de fonte aceitos: Manual Tecnico, Instrucoes de Uso/IFU, Nota Fiscal, Plano de Contas, Folder/Catalogo, Website do Fabricante
- RN-UC007-002: Cada tipo de fonte tem extensoes aceitas especificas (.pdf, .doc, .docx para manuais; .pdf, .xlsx, .xls, .csv para NFs)
- RN-UC007-003: Fonte "Website do Fabricante" nao requer upload de arquivo, apenas URL
- RN-UC007-004: A IA recebe prompt contextualizado por tipo de documento (ex: "Cadastre este produto a partir do manual")
- RN-UC007-005: O usuario pode opcionalmente informar nome do produto, area, classe e subclasse antes do upload

## Componentes de UI envolvidos
- PortfolioPage.tsx — aba `Cadastro por IA`
- Seletor `Tipo de Documento` (UPLOAD_TYPES com 6 opcoes)
- TextInput para nome do produto (uploadNomeProduto) — opcional
- TextInput para URL do website (websiteUrl) — visivel apenas quando tipo === "website"
- Input file com accept filtrado por tipo de documento
- SelectInput para Area (uploadArea), Classe (uploadClasse), Subclasse (uploadSubclasse) — opcionais; quando Subclasse e informada, e enviada para melhorar extracao conforme mascara tecnica
- Botao `Processar com IA` — cria sessao `cadastro-produto` e dispara o processamento
- Card de feedback com resposta textual da IA (ReactMarkdown)
- Loader2 animado durante processamento

## Endpoints ou servicos envolvidos
- POST /api/chat/session (via `createSession`) — cria sessao `cadastro-produto`
- POST /api/chat/message (via `sendMessageWithFile`) — envia arquivo + prompt especifico por tipo para IA (usado em manual, instrucoes, folders, nfs, plano_contas)
- POST /api/chat/message (via `sendMessage`) — envia URL + prompt para IA (usado em website)
- GET /api/produtos (via `fetchProdutos()`) — recarrega lista apos cadastro

## Dados manipulados
- Arquivo PDF/DOC/DOCX/XLSX/XLS/CSV do documento tecnico
- URL do website do fabricante (quando aplicavel)
- Nome do produto (opcional, preenchido pelo usuario)
- Area, classe e subclasse (opcionais, preenchidos pelo usuario)
- Produto: id, nome, fabricante, modelo, ncm, registro_anvisa, descricao, area_id, classe_id, subclasse_id
- Especificacoes: campo, valor, unidade, tipo
- Score de completude: percentual 0-100%

## Fluxo principal

### Passo 1: Acessar aba Cadastro por IA
**Acao do ator:** O usuario clica na aba "Cadastro por IA" na PortfolioPage.
**Elemento UI:** Tab button "Cadastro por IA" com icone Sparkles.
**Resposta visual:** O sistema exibe o formulario de cadastro assistido com os campos de tipo de documento, nome do produto, selecao de hierarquia e area de upload.
**Efeito funcional:** Estado activeTab muda para "cadastroIA".
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot da aba Cadastro por IA com formulario visivel.

### Passo 2: Selecionar tipo de documento
**Acao do ator:** O usuario seleciona o tipo de documento no dropdown (ex: "Manual Tecnico").
**Elemento UI:** SelectInput com opcoes: Manual Tecnico, Instrucoes de Uso/IFU, Nota Fiscal, Plano de Contas, Folder/Catalogo, Website do Fabricante.
**Resposta visual:** O campo de upload atualiza o filtro de extensoes aceitas (accept) de acordo com o tipo selecionado. Se tipo "Website", campo de URL aparece no lugar do input file.
**Efeito funcional:** Estado tipoDocumento atualizado. O prompt que sera enviado a IA e contextualizado (ex: "Cadastre este produto a partir do manual").
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot mostrando tipo selecionado e campo de upload com extensoes corretas.

### Passo 3: Informar nome do produto (opcional)
**Acao do ator:** O usuario digita o nome do produto no campo "Nome do Produto".
**Elemento UI:** TextInput (uploadNomeProduto).
**Resposta visual:** Texto aparece no campo.
**Efeito funcional:** Estado uploadNomeProduto atualizado.
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot com nome do produto preenchido.

### Passo 4: Selecionar hierarquia (opcional)
**Acao do ator:** O usuario seleciona area, classe e subclasse nos dropdowns correspondentes.
**Elemento UI:** Tres SelectInputs encadeados — area filtra classes, classe filtra subclasses.
**Resposta visual:** Cada select e populado com opcoes do backend. Selecionar area filtra as classes visiveis; selecionar classe filtra as subclasses.
**Efeito funcional:** Estados uploadArea, uploadClasse, uploadSubclasse atualizados.
**Requisicao de rede:** Dados ja carregados na inicializacao (crudList areas-produto, classes-produto-v2, subclasses-produto).
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot com hierarquia selecionada.

### Passo 5: Fazer upload do arquivo
**Acao do ator:** O usuario clica no campo de upload e seleciona o arquivo do documento tecnico.
**Elemento UI:** Input file (fileInputRef) com accept filtrado pelo tipo de documento.
**Resposta visual:** Nome do arquivo aparece no campo de upload.
**Efeito funcional:** Estado uploadFile recebe o File selecionado.
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot mostrando arquivo selecionado.

### Passo 6: Processar com IA
**Acao do ator:** O usuario clica no botao `Processar com IA`.
**Elemento UI:** Botao `Processar com IA`.
**Resposta visual:** Sistema exibe Loader2 animado e mensagem de processamento. Botao fica desabilitado durante o processamento.
**Efeito funcional:** Sistema cria sessao `cadastro-produto`. Para `website`: usa `sendMessage(session_id, "Busque produtos no website ... e cadastre")`. Para `manual`, `instrucoes`, `folders`, `nfs` e `plano_contas`: escolhe prompt especifico por tipo e envia `sendMessageWithFile(session_id, prompt, arquivo, subclasse_id?)`.
**Requisicao de rede:** POST /api/chat/session + POST /api/chat/message com multipart/form-data (arquivo + prompt) ou apenas mensagem (website).
**Persistencia:** IA processa documento, extrai especificacoes e cadastra produto(s) no banco.
**Evidencia esperada:** Screenshot do estado de loading com mensagem de processamento.

### Passo 7: Receber resposta da IA
**Acao do ator:** O usuario aguarda o processamento.
**Elemento UI:** Card de resposta com ReactMarkdown renderizando a resposta textual da IA inline na pagina.
**Resposta visual:** Loader desaparece. Retorno textual da IA e mostrado inline. Mensagem de sucesso.
**Efeito funcional:** Resposta da IA exibida inline. A tela executa `fetchProdutos()` para recarregar a grade de produtos.
**Requisicao de rede:** GET /api/produtos (via `fetchProdutos()`).
**Persistencia:** Produto(s) salvo(s) no banco com especificacoes extraidas.
**Evidencia esperada:** Screenshot da resposta da IA com dados do produto extraido.

### Passo 8: Verificar produto na lista
**Acao do ator:** O usuario navega para a aba "Produtos" para verificar o produto criado.
**Elemento UI:** Tab "Produtos" + DataTable de produtos.
**Resposta visual:** Novo produto aparece na tabela com campos preenchidos pela IA. Score de completude exibido via ScoreBar.
**Efeito funcional:** Produto presente na lista filtrada.
**Requisicao de rede:** GET /api/produtos (ja recarregado).
**Persistencia:** Produto persistido.
**Evidencia esperada:** Screenshot da lista de produtos com o novo produto visivel e score de completude.

## Fluxos alternativos

### FA-001: Upload por Website (URL)
**Condicao:** Tipo de documento selecionado e "Website do Fabricante".
**Variacao no Passo 5:** Em vez de input file, o usuario preenche o campo TextInput de URL (websiteUrl). O campo de upload de arquivo e ocultado.
**Variacao no Passo 6:** O prompt enviado a IA inclui a URL em vez de arquivo. sendMessage e usado em vez de sendMessageWithFile.

### FA-002: Upload de Nota Fiscal (multiplos produtos)
**Condicao:** Tipo de documento selecionado e "Nota Fiscal (NFS)" ou "Plano de Contas (ERP)".
**Variacao no Passo 7:** A IA extrai e cadastra multiplos produtos de uma vez. A resposta lista todos os produtos criados. A tabela de produtos mostra multiplas novas entradas.

### FA-003: Upload sem hierarquia
**Condicao:** Usuario nao seleciona area/classe/subclasse.
**Variacao:** A IA tenta inferir a classificacao a partir do conteudo do documento. O produto pode ser criado sem hierarquia definida, ficando como "Sem classificacao".

## Fluxos de excecao

### FE-001: Arquivo nao selecionado
**Condicao:** Usuario clica "Enviar para IA" sem selecionar arquivo (e tipo != website).
**Resposta:** Sistema exibe mensagem de erro solicitando selecao de arquivo.

### FE-002: Erro de processamento da IA
**Condicao:** DeepSeek retorna erro ou timeout.
**Resposta:** Sistema exibe mensagem de erro no card de feedback. Botao de envio e reabilitado para nova tentativa.

### FE-003: Formato de arquivo nao suportado
**Condicao:** Arquivo selecionado nao corresponde ao accept do tipo de documento.
**Resposta:** Input file rejeita a selecao pelo atributo accept do HTML.

### FE-004: Backend indisponivel
**Condicao:** Backend na porta 5007 esta fora do ar.
**Resposta:** Requisicao falha. Mensagem de erro de rede exibida no card de feedback.

## Criterios de validacao
1. Upload de arquivo PDF deve disparar chamada POST /api/chat/message com multipart
2. IA deve retornar dados estruturados do produto (nome, fabricante, modelo, NCM, specs)
3. Produto deve ser persistido no banco e visivel na listagem
4. Score de completude deve ser calculado e exibido
5. Seis tipos de documento devem estar disponiveis no select
6. Tipo "Website" deve mostrar campo URL em vez de file input
7. Tipo "NF" e "Plano de Contas" devem suportar criacao de multiplos produtos

## Implementacao Observada

**Status:** IMPLEMENTADO

**Fonte:** UC-F07 (Cadastrar produto por IA a partir de manual, IFU, folder, NFS, plano de contas ou website) do documento `CASOS DE USO EMPRESA PORTFOLIO PARAMETRIZACAO.md`.

**Observacoes criticas:**
- O botao real chama-se `Processar com IA` (nao "Enviar para IA").
- O sistema cria uma sessao de tipo `cadastro-produto` antes de enviar o prompt.
- Para `website`, usa `sendMessage` (sem arquivo); para os demais tipos, usa `sendMessageWithFile`.
- Quando a `Subclasse` e informada, ela e enviada no prompt para melhorar a extracao conforme a mascara tecnica. Para `website`, a classificacao opcional por subclasse **nao** e enviada no prompt atual.
- `nfs` e `plano_contas` admitem extracao de **multiplos itens** de um unico arquivo.
- Cada tipo de origem tem um **prompt especifico** diferente (nao um prompt generico).
- O retorno textual da IA e mostrado inline na pagina (nao em uma pagina separada).
- Apos o processamento, a tela executa `fetchProdutos()` automaticamente para refletir os novos produtos.
