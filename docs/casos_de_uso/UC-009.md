# Caso de Uso: UC-009

## Nome
Hierarquia de Classes e Subclasses

## Objetivo
Permitir que o usuario gerencie a hierarquia de classificacao de produtos: criar, editar e excluir areas, classes e subclasses, incluindo associacao de NCMs e definicao de campos mascara para especificacoes tecnicas.

## Escopo
Modulo Portfolio — aba "Classificacao" (PortfolioPage.tsx) e Modulo Parametrizacoes (ParametrizacoesPage.tsx)

## Atores
- Administrador / Usuario (operador do sistema)

## Gatilho
O usuario acessa a aba "Classificacao" na PortfolioPage ou a secao de "Classes de Produto" na ParametrizacoesPage.

## Pre-condicoes
1. Usuario autenticado no sistema
2. Backend ativo na porta 5007
3. Tabelas areas-produto, classes-produto-v2 e subclasses-produto disponiveis no backend

## Pos-condicoes
1. Hierarquia Area -> Classe -> Subclasse criada/atualizada no banco de dados
2. NCMs associados a classes e subclasses
3. Campos mascara definidos para subclasses (estrutura de especificacoes tecnicas)
4. Hierarquia disponivel para uso no cadastro e classificacao de produtos

## Regras de negocio
- RN-PORT-001: Produtos organizados em hierarquia Area -> Classe -> Subclasse
- RN-UC009-001: Uma area pode conter multiplas classes
- RN-UC009-002: Uma classe pertence a exatamente uma area (campo area_id)
- RN-UC009-003: Uma classe pode conter multiplas subclasses
- RN-UC009-004: Uma subclasse pertence a exatamente uma classe (campo classe_id)
- RN-UC009-005: Classes e subclasses possuem NCMs associados (array de codigos NCM)
- RN-UC009-006: Subclasses podem ter campos_mascara — JSON que define os campos de especificacao tecnica com: campo, tipo (texto/numero/decimal/select/boolean), unidade, placeholder, opcoes, obrigatorio
- RN-UC009-007: Classes na ParametrizacoesPage sao gerenciadas como tipo "area" | "classe" | "subclasse" via campo tipo na API classes-produto-v2
- RN-UC009-008: Ao excluir uma classe, suas subclasses e produtos associados devem ser tratados

## Componentes de UI envolvidos

### Na PortfolioPage (aba "Classificacao"):
- Tab "Classificacao" (activeTab === "classificacao")
- Arvore hierarquica com areas expandiveis (expandedArea)
- Subitens de classes expandiveis (expandedClasse)
- Sub-subitens de subclasses
- Icones ChevronDown/ChevronRight para expansao
- Badges com contagem de produtos por classe/subclasse (qtd_produtos)

### Na ParametrizacoesPage (secao Classes):
- Lista de classes com tipo (area/classe/subclasse)
- Campos: nome, tipo, NCMs (array), classe_pai_id, campos_mascara
- Formularios de criacao/edicao de classe
- CampoMascara editor: campo (nome) e placeholder

## Endpoints ou servicos envolvidos
- CRUD /api/crud/areas-produto — gestao de areas
- CRUD /api/crud/classes-produto-v2 — gestao de classes (com tipo, ncms, campos_mascara, area_id)
- CRUD /api/crud/subclasses-produto — gestao de subclasses (com classe_id, ncms, campos_mascara)
- GET /api/crud/areas-produto?limit=100 — lista todas as areas
- GET /api/crud/classes-produto-v2?limit=200 — lista todas as classes
- GET /api/crud/subclasses-produto?limit=500 — lista todas as subclasses
- POST /api/crud/areas-produto — criar area
- PUT /api/crud/areas-produto/{id} — editar area
- DELETE /api/crud/areas-produto/{id} — excluir area
- POST /api/crud/classes-produto-v2 — criar classe
- PUT /api/crud/classes-produto-v2/{id} — editar classe
- DELETE /api/crud/classes-produto-v2/{id} — excluir classe
- POST /api/crud/subclasses-produto — criar subclasse
- PUT /api/crud/subclasses-produto/{id} — editar subclasse
- DELETE /api/crud/subclasses-produto/{id} — excluir subclasse

## Dados manipulados
- Area: id, nome
- Classe: id, nome, tipo ("area"|"classe"|"subclasse"), ncms (string[]), area_id, campos_mascara (JSON), qtd_produtos
- Subclasse: id, nome, ncms (string), classe_id, campos_mascara (CampoMascara[] | null), produtos (count)
- CampoMascara: campo (nome do campo), tipo, unidade, placeholder, opcoes, obrigatorio

## Fluxo principal

### Passo 1: Acessar aba Classificacao
**Acao do ator:** O usuario clica na aba "Classificacao" na PortfolioPage.
**Elemento UI:** Tab button "Classificacao" com icone FolderOpen.
**Resposta visual:** Sistema exibe a arvore hierarquica de areas, classes e subclasses. Cada area e exibida como item expandivel. Numero de produtos por nivel e exibido.
**Efeito funcional:** Estado activeTab muda para "classificacao". Dados de areasBackend, classesV2Backend e subclassesBackend ja carregados na inicializacao da pagina.
**Requisicao de rede:** Nenhuma (dados carregados no useEffect inicial: crudList areas-produto, classes-produto-v2, subclasses-produto).
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot da arvore hierarquica com areas listadas.

### Passo 2: Expandir area
**Acao do ator:** O usuario clica em uma area para expandi-la e ver suas classes.
**Elemento UI:** Item de area com icone ChevronRight (fechado) ou ChevronDown (aberto).
**Resposta visual:** Area expande e exibe suas classes filhas. Cada classe mostra nome, NCMs associados e numero de produtos.
**Efeito funcional:** Estado expandedArea recebe o id da area clicada.
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot da area expandida com classes visiveis.

### Passo 3: Expandir classe e ver subclasses
**Acao do ator:** O usuario clica em uma classe para expandi-la e ver suas subclasses.
**Elemento UI:** Item de classe com icone ChevronRight/ChevronDown.
**Resposta visual:** Classe expande e exibe suas subclasses filhas. Cada subclasse mostra nome, NCMs e contagem de produtos.
**Efeito funcional:** Estado expandedClasse recebe o id da classe clicada.
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot da classe expandida com subclasses visiveis.

### Passo 4: Criar nova area (via Parametrizacoes)
**Acao do ator:** O usuario acessa ParametrizacoesPage > secao Classes e clica em "Nova Area" ou equivalente.
**Elemento UI:** Formulario de criacao com campo nome.
**Resposta visual:** Formulario de nova area abre. Campo nome exibido.
**Efeito funcional:** Formulario de criacao inicializado.
**Requisicao de rede:** Nenhuma.
**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot do formulario de criacao de area.

### Passo 5: Preencher dados e salvar area
**Acao do ator:** O usuario preenche o nome da area e clica em "Salvar".
**Elemento UI:** TextInput para nome + ActionButton "Salvar".
**Resposta visual:** Area criada com sucesso. Lista atualizada com nova area.
**Efeito funcional:** crudCreate("areas-produto", { nome }) chamado.
**Requisicao de rede:** POST /api/crud/areas-produto com { nome }.
**Persistencia:** Area salva no banco de dados.
**Evidencia esperada:** Screenshot da lista atualizada com nova area.

### Passo 6: Criar classe dentro da area
**Acao do ator:** O usuario cria uma nova classe associada a area recem-criada.
**Elemento UI:** Formulario de criacao de classe com campos: nome, area_id (select), NCMs (input de tags/texto).
**Resposta visual:** Formulario de classe exibido com select de area preenchido.
**Efeito funcional:** crudCreate("classes-produto-v2", { nome, area_id, tipo: "classe", ncms }) chamado.
**Requisicao de rede:** POST /api/crud/classes-produto-v2 com dados da classe.
**Persistencia:** Classe salva no banco de dados associada a area.
**Evidencia esperada:** Screenshot da classe criada na hierarquia.

### Passo 7: Criar subclasse com NCMs e campos mascara
**Acao do ator:** O usuario cria uma subclasse dentro da classe, definindo NCMs e campos mascara.
**Elemento UI:** Formulario de criacao de subclasse com campos: nome, classe_id (select), NCMs (texto), campos_mascara (editor JSON ou formulario estruturado com campo e placeholder por CampoMascara).
**Resposta visual:** Formulario exibe editor de campos mascara onde cada campo tem: nome do campo e placeholder.
**Efeito funcional:** crudCreate("subclasses-produto", { nome, classe_id, ncms, campos_mascara }) chamado.
**Requisicao de rede:** POST /api/crud/subclasses-produto com dados da subclasse incluindo campos_mascara como JSON.
**Persistencia:** Subclasse salva no banco de dados com campos mascara. Estes campos definirao a estrutura de especificacoes tecnicas dos produtos desta subclasse.
**Evidencia esperada:** Screenshot da subclasse criada com campos mascara definidos.

### Passo 8: Verificar hierarquia completa

**Elemento UI:** Arvore hierarquica atualizada.

**Persistencia:** Nenhuma.
**Evidencia esperada:** Screenshot da hierarquia completa com novos itens.

## Fluxos alternativos

### FA-001: Editar classe existente
**Condicao:** Usuario seleciona uma classe existente para edicao.
**Variacao:** Formulario de edicao preenchido com dados atuais. crudUpdate usado em vez de crudCreate. NCMs e campos_mascara editaveis.

### FA-002: Excluir subclasse
**Condicao:** Usuario solicita exclusao de uma subclasse.
**Variacao:** Sistema verifica se ha produtos vinculados. Se houver, exibe alerta. crudDelete("subclasses-produto", id) chamado.

### FA-003: Gerenciar hierarquia via ParametrizacoesPage
**Condicao:** Usuario utiliza a ParametrizacoesPage para gestao completa de classes.
**Variacao:** Interface de lista tabular com filtros por tipo (area/classe/subclasse). CRUD completo via formularios modais.

## Fluxos de excecao

### FE-001: Nome duplicado
**Condicao:** Usuario tenta criar area/classe/subclasse com nome ja existente.
**Resposta:** Backend retorna erro de unicidade. Mensagem de erro exibida no formulario.

### FE-002: Exclusao com produtos vinculados
**Condicao:** Usuario tenta excluir classe/subclasse com produtos associados.
**Resposta:** Sistema alerta sobre produtos vinculados. Acao bloqueada ou solicita confirmacao.

### FE-003: Campos mascara com formato invalido
**Condicao:** JSON de campos_mascara mal formado.
**Resposta:** parseMascaraTop() retorna array vazio. Especificacoes nao sao renderizadas.

## Criterios de validacao
1. Arvore hierarquica deve exibir areas > classes > subclasses corretamente
2. Criar area deve persistir via POST /api/crud/areas-produto
3. Criar classe deve associar a area via area_id
4. Criar subclasse deve associar a classe via classe_id e aceitar campos_mascara
5. NCMs devem ser exibidos e editaveis em classes e subclasses
6. campos_mascara devem ser parseados corretamente por parseMascaraTop()
7. Exclusao deve tratar produtos vinculados
