# Caso de Uso: UC-042

## Nome
Registrar resultado (vitoria/derrota)

## Objetivo
Permitir ao usuario registrar o resultado final de uma licitacao cujo edital foi submetido, classificando como vitoria, derrota ou cancelado, com captura de valor final, vencedor e motivos, gerando automaticamente contrato em caso de vitoria.

## Escopo
Modulo Follow-up — FollowupPage.tsx

## Atores
- Usuario autenticado (Analista Comercial)
- Sistema backend (Flask)
- Modulo de contratos (criacao automatica)
- Modulo de notificacoes (alertas de resultado)

## Gatilho
Usuario clica no botao "Registrar" em um edital da lista de pendentes na aba Resultados da FollowupPage.

## Pre-condicoes
1. Usuario esta autenticado no sistema (editais_ia_access_token)
2. Edital foi submetido previamente (UC-031) e aparece na lista de pendentes
3. Backend Flask esta ativo na porta 5007
4. Edital possui status "submetido", "proposta_enviada" ou "em_pregao"
5. Dados do edital incluem numero, orgao e valor de referencia
6. Endpoint POST /api/followup/registrar-resultado esta disponivel
7. Empresa do usuario cadastrada com dados validos

## Pos-condicoes
1. Resultado registrado no banco de dados com tipo (vitoria/derrota/cancelado)
2. Status do edital atualizado para "ganho", "perdido" ou "cancelado"
3. Em caso de vitoria: contrato criado automaticamente via ProducaoPage
4. StatCards atualizados com novo total de vitorias, derrotas e taxa de sucesso
5. Edital removido da lista de pendentes e adicionado a lista de resultados
6. Valor final, vencedor e observacoes persistidos
7. Historico de resultados disponivel para analise de desempenho

## Regras de negocio
1. **RN-FUP-001:** Tres tipos de resultado possiveis: vitoria, derrota, cancelado
2. **RN-FUP-002:** Em vitoria, campo valor_final e obrigatorio para criacao automatica do contrato
3. **RN-FUP-003:** Em derrota, campo vencedor e motivo_derrota sao solicitados para inteligencia competitiva
4. **RN-FUP-004:** Em cancelamento, campo justificativa e obrigatorio
5. **RN-FUP-005:** Taxa de sucesso = vitorias / (vitorias + derrotas) * 100, cancelados nao entram no calculo
6. **RN-FUP-006:** Contrato criado automaticamente em vitoria herda dados do edital (numero, orgao, valor)
7. **RN-FUP-007:** Resultado e irreversivel apos registro — edital nao pode voltar para pendentes

## Componentes de UI envolvidos
1. TabPanel com abas "Resultados" e "Alertas" (TabPanel component)
2. StatCards com 4 metricas: Pendentes, Vitorias, Derrotas, Taxa de Sucesso (Card component)
3. DataTable de pendentes com colunas: Edital, Orgao, Data Submissao, Valor Proposta, Acao
4. DataTable de resultados com colunas: Edital, Orgao, Resultado (badge colorido), Valor Final, Data
5. Modal de registro com campos condicionais por tipo de resultado (Modal component)
6. SelectInput para tipo de resultado (vitoria/derrota/cancelado)
7. FormFields condicionais: valor_final, vencedor, motivo, justificativa, observacoes

## Endpoints ou servicos envolvidos
1. **GET /api/followup/pendentes** — lista editais com proposta enviada aguardando resultado
2. **GET /api/followup/resultados** — lista editais ja com resultado registrado
3. **POST /api/followup/registrar-resultado** — registra resultado com tipo, valor, vencedor e motivos
4. **POST /api/crud/contratos** — criacao automatica de contrato em caso de vitoria
5. **GET /api/alertas-vencimento/regras** — regras de alerta configuradas
6. **GET /api/alertas-vencimento/consolidado** — vencimentos consolidados
7. **PUT /api/editais/{id}** — atualizacao de status do edital

## Dados manipulados
1. **EditalPendente:** id, numero, orgao, data_abertura, valor_referencia, status, created_at, updated_at
2. **EditalResultado:** id, numero, orgao, status, valor_referencia, updated_at
3. **Payload de registro:** edital_id, tipo, valor_final, vencedor, motivo_derrota, observacoes
4. **Tipo resultado:** "vitoria" | "derrota" | "cancelado" (mapeado para "ganho"/"perdido"/"cancelado")
5. **StatCards:** pendentes.length, vitorias count, derrotas count, taxa % calculada
6. **Badge colorido:** ganho=#16a34a, perdido=#dc2626, cancelado=#6b7280
7. **Contrato gerado:** numero_contrato, orgao, objeto, valor_total, status="vigente"

## Fluxo principal
1. Usuario acessa FollowupPage e visualiza aba "Resultados" com StatCards (Pendentes, Vitorias, Derrotas, Taxa) e DataTable de editais pendentes
2. Usuario identifica um edital na lista de pendentes e clica no botao "Registrar" na coluna Acao
3. Sistema abre Modal com SelectInput de tipo (vitoria/derrota/cancelado) e campo valor_final pre-preenchido com valor_referencia
4. Usuario seleciona tipo "vitoria", confirma valor_final e preenche observacoes opcionais
5. Usuario clica em "Confirmar" e sistema envia POST /api/followup/registrar-resultado com payload completo
6. Backend persiste resultado, atualiza status do edital para "ganho" e cria contrato automaticamente
7. Frontend fecha Modal, recarrega fetchData() e StatCards atualizam com nova taxa de sucesso

## Fluxos alternativos
1. **FA-01 — Registro de derrota:** Usuario seleciona "derrota"; campos vencedor e motivo_derrota aparecem; sistema persiste dados para analise competitiva
2. **FA-02 — Cancelamento de licitacao:** Usuario seleciona "cancelado"; campo justificativa aparece; cancelados nao afetam taxa de sucesso
3. **FA-03 — Valor final diferente do proposto:** Usuario altera valor_final no Modal; sistema registra diferenca para analise de margem
4. **FA-04 — Nenhum edital pendente:** Lista de pendentes vazia; sistema exibe mensagem informativa sem botoes de acao
5. **FA-05 — Edital com valor de referencia nulo:** Campo valor_final inicia vazio; usuario preenche manualmente
6. **FA-06 — Consulta de historico:** Usuario visualiza DataTable de resultados com badges coloridos (verde/vermelho/cinza) e filtra por periodo
7. **FA-07 — Registro via Chat:** Usuario informa resultado pelo FloatingChat; DeepSeek roteia para tool equivalente

## Fluxos de excecao
1. **FE-01 — Erro de rede no POST:** Frontend captura erro no try/catch e exibe mensagem no console; Modal permanece aberto para retry
2. **FE-02 — Backend retorna erro 4xx:** Edital ja teve resultado registrado anteriormente; frontend recebe erro e exibe alerta
3. **FE-03 — Token expirado:** Requisicao retorna 401; sistema redireciona para LoginPage
4. **FE-04 — Edital nao encontrado:** Backend retorna 404; frontend exibe mensagem de edital invalido
5. **FE-05 — Valor final invalido:** parseFloat retorna NaN; backend rejeita com erro de validacao
6. **FE-06 — Erro ao criar contrato automatico:** Vitoria registrada mas contrato falha; sistema registra resultado e loga erro
7. **FE-07 — Concorrencia:** Dois usuarios tentam registrar resultado do mesmo edital; primeiro registro prevalece, segundo recebe erro

### Sequencia de eventos real (UC-FU01)
1. Usuario acessa FollowupPage — aba "Aguardando" ativa por padrao
2. Sistema carrega stats FU01-F01 a FU01-F04
3. FU01-F05 exibe editais com status "submetido" ordenados por dias aguardando (decrescente)
4. Usuario clica [R] Registrar em um edital — modal abre com dados pre-preenchidos
5. Sistema exibe edital, orgao e valor proposto no cabecalho do modal
6. Usuario seleciona tipo em FU01-F08: Vitoria, Derrota ou Cancelado

8. Se Derrota: mostra FU01-F09, FU01-F10 Empresa Vencedora e FU01-F11 Motivo

10. Usuario preenche campos obrigatorios e confirma checkboxes FU01-F13 e FU01-F14
11. Usuario clica FU01-F15 — backend processa via `tool_registrar_resultado`
12. Backend atualiza status do edital, alimenta preco_historico e concorrentes
13. Modal fecha; FU01-F07 atualiza com novo registro; FU01-F05 remove o edital
14. Se vitoria, toast sugere criacao de contrato: "Deseja criar contrato? [Ir para Producao]"
