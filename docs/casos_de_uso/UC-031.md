# Caso de Uso: UC-031

## Nome
Submeter proposta

## Objetivo
Permitir ao usuario gerenciar a submissao da proposta ao portal de licitacao, incluindo checklist pre-submissao, tracking de status, fracionamento de PDF (Smart Split) para portais com limite de tamanho, e registro do protocolo de envio.

## Escopo
Modulo Submissao — SubmissaoPage.tsx

## Atores
- Usuario autenticado (Analista Comercial / Operador de Licitacao)
- Sistema backend (Flask)
- Portal externo (ComprasNet / gov.br)
- Banco de dados

## Gatilho
Usuario acessa a SubmissaoPage com proposta em status "aprovada", pronta para envio ao portal de licitacao.

## Pre-condicoes
1. Proposta em status "aprovada" com texto tecnico e precos definidos (UC-026 a UC-029)
2. Dossie exportado em formato adequado (UC-030 executado, recomendado)
3. Checklist de documentos obrigatorios verificado (auditoria documental UC-029)
4. Backend Flask ativo e banco de dados acessivel
5. Dados do edital incluindo data e hora de abertura disponiveis
6. Link do portal de licitacao disponivel (ComprasNet, gov.br)
7. Usuario com credenciais de acesso ao portal externo

## Pos-condicoes
1. Checklist pre-submissao verificado com todos os itens marcados
2. Status da submissao atualizado para "enviada" no banco
3. Protocolo de submissao registrado com numero e timestamp
4. Proposta com status alterado de "aprovada" para "enviada"
5. Dashboard atualizado com edital em status "submetido"
6. Historico de submissao registrado para auditoria
7. Edital pronto para acompanhamento de disputa (proximo passo do fluxo)

## Regras de negocio
1. **RN-SUB-001:** Checklist valida completude documental antes da submissao
2. **RN-SUB-002:** Smart Split respeita limites de tamanho dos portais (tipicamente 5-25MB por arquivo)
3. **RN-SUB-003:** Status de submissao: "aguardando" -> "enviada" -> "aprovada" (pelo portal)
4. **RN-SUB-004:** Submissao e assistida (checklist + link externo); portal nao tem API de submissao
5. **RN-SUB-005:** Protocolo de submissao e obrigatorio apos envio para comprovacao
6. **RN-SUB-006:** Data/hora de abertura definem deadline de submissao com alerta visual
7. **RN-SUB-007:** Todos os itens do checklist devem estar marcados para habilitar botao de registro

## Componentes de UI envolvidos
1. DataTable de propostas prontas (PropostaPronta) com colunas: edital, orgao, produto, valor, status, data/hora abertura
2. Card de checklist pre-submissao com items: propostaTecnica, precoDefinido, documentosAnexados, revisaoFinal
3. Checkbox para cada item do checklist com label descritivo e icone (CheckSquare)
4. Botao "Abrir Portal" (ActionButton com icone ExternalLink) que abre link do portal externo
5. Botao "Smart Split PDF" (ActionButton com icone Upload) para fracionar PDF grande
6. Modal de registro de protocolo com FormField para numero_protocolo e data_envio
7. StatusBadge com cores: "aguardando" (amarelo), "enviada" (azul), "aprovada" (verde)

## Endpoints ou servicos envolvidos
1. **GET /api/submissoes** — lista submissoes pendentes e realizadas
2. **POST /api/submissoes** — cria registro de submissao com dados do checklist
3. **PUT /api/submissoes/{id}/status** — atualiza status da submissao
4. **POST /api/submissoes/{id}/split-pdf** — fraciona PDF para limites do portal
5. **crudList('propostas', { status: 'aprovada' })** — lista propostas prontas para submissao
6. **crudUpdate('propostas', id, { status: 'enviada' })** — atualiza status da proposta
7. **crudCreate('submissao_logs')** — registra log de submissao com protocolo

## Dados manipulados
1. **PropostaPronta:** id, edital, orgao, produto, valor, dataAbertura, horaAbertura, status, checklist
2. **Checklist:** propostaTecnica (boolean), precoDefinido (boolean), documentosAnexados (num/total), revisaoFinal (boolean)
3. **Status submissao:** "aguardando"|"enviada"|"aprovada" — estado da submissao
4. **Protocolo:** string — numero de protocolo fornecido pelo portal apos envio
5. **Data envio:** datetime — data e hora do registro de envio
6. **Link portal:** string — URL do portal de licitacao (ComprasNet, gov.br)
7. **Smart Split:** arquivo original e partes fracionadas com tamanhos

## Fluxo principal
1. Usuario acessa SubmissaoPage e sistema carrega DataTable de PropostaPronta com propostas em status "aprovada"
2. Usuario seleciona uma proposta e sistema exibe Card de checklist com items de verificacao pre-submissao
3. Usuario marca cada item do checklist: proposta tecnica OK, preco definido, documentos anexados, revisao final
4. Sistema valida que todos os itens do checklist estao marcados e habilita botao "Abrir Portal" e "Registrar Envio"
5. Usuario clica "Abrir Portal" que abre link externo do ComprasNet/gov.br em nova aba para submissao manual
6. Apos envio no portal externo, usuario retorna e clica "Registrar Envio" abrindo Modal de protocolo
7. Usuario preenche numero de protocolo e confirma; sistema atualiza status para "enviada" e registra log

## Fluxos alternativos
1. **FA-01 — Smart Split:** PDF excede limite do portal; usuario clica "Smart Split" para fracionar em partes menores
2. **FA-02 — Submissao parcial:** Portal aceita envio em etapas; usuario registra protocolo parcial
3. **FA-03 — Portal alternativo:** Edital em plataforma diferente (Licitacoes-e, BEC); sistema exibe link adequado
4. **FA-04 — Resubmissao:** Proposta rejeitada pelo portal; usuario corrige e submete novamente
5. **FA-05 — Prazo critico:** Data de abertura proxima (< 24h); sistema exibe alerta Clock vermelho
6. **FA-06 — Checklist parcial:** Documento nao obrigatorio ausente; usuario marca como "dispensado" e prossegue
7. **FA-07 — Envio antecipado:** Usuario submete com antecedencia; sistema registra e monitora status

## Fluxos de excecao
1. **FE-01 — Checklist incompleto:** Itens nao marcados; botao "Registrar Envio" desabilitado com tooltip
2. **FE-02 — Proposta nao aprovada:** Status diferente de "aprovada"; sistema impede submissao
3. **FE-03 — Portal indisponivel:** Link do portal retorna erro; sistema exibe mensagem e link alternativo
4. **FE-04 — Protocolo invalido:** Numero de protocolo vazio ou com formato invalido; sistema exibe erro
5. **FE-05 — Prazo expirado:** Data de abertura ja passou; sistema bloqueia submissao com mensagem
6. **FE-06 — Erro no Smart Split:** Fracionamento do PDF falha; sistema exibe erro e sugere reducao manual
7. **FE-07 — Erro de persistencia:** PUT retorna erro 500; frontend exibe toast de erro e mantem dados
