# Caso de Uso: UC-052

## Nome
Designar gestor e fiscal

## Objetivo
Permitir ao usuario designar gestor e fiscal de contrato conforme exigencia da Lei 14.133 Art. 117, com validacao de que a mesma pessoa nao pode exercer ambas as funcoes simultaneamente (§5o), incluindo dados de portaria e periodo de vigencia.

## Escopo
Modulo Contratos — ProducaoPage.tsx > Aba Gestor/Fiscal

## Atores
- Usuario autenticado (Gestor de Contratos / Autoridade Administrativa)
- Sistema backend (Flask)
- Motor de validacao legal (Art. 117 Lei 14.133)

## Gatilho
Usuario seleciona um contrato na aba Contratos da ProducaoPage e acessa a aba de designacoes para registrar gestor e/ou fiscal.

## Pre-condicoes
1. Usuario esta autenticado no sistema
2. Contrato cadastrado e selecionado (UC-048)
3. Backend Flask ativo com endpoint POST /api/contratos/{id}/designacoes
4. Endpoint GET /api/contratos/{id}/designacoes retorna designacoes existentes
5. Dados do designado disponiveis (nome, cargo, portaria)
6. Conhecimento das exigencias legais (Art. 117 Lei 14.133)
7. ProducaoPage acessivel no menu do sistema

## Pos-condicoes
1. Designacao persistida com tipo (gestor/fiscal), nome, cargo, portaria e periodo
2. Validacao de que gestor e fiscal nao sao a mesma pessoa
3. Designacao marcada como ativa (boolean) com periodo de vigencia
4. DataTable de designacoes atualizada com novo registro
5. Contrato em conformidade com Art. 117 (gestor + fiscal obrigatorios)
6. Designacao anterior pode ser desativada ao registrar nova
7. Modal de cadastro fechado e formulario resetado

## Regras de negocio
1. **RN-CON-003:** Obrigatorio designar gestor e fiscal para cada contrato (Art. 117 Lei 14.133)
2. **RN-CON-004:** Mesma pessoa NAO pode ser gestor e fiscal simultaneamente (§5o Art. 117)
3. **RN-DES-001:** Tipo de designacao: "gestor" ou "fiscal"
4. **RN-DES-002:** Campos obrigatorios: tipo, nome, cargo, portaria_numero
5. **RN-DES-003:** Periodo de vigencia: data_inicio e data_fim (opcional)
6. **RN-DES-004:** Designacao possui flag "ativo" (boolean) para controle de vigencia
7. **RN-DES-005:** Historico de designacoes mantido (desativadas ficam no registro)

## Componentes de UI envolvidos
1. TabPanel aba "Gestor/Fiscal" com icone Users na ProducaoPage
2. ActionButton "Nova Designacao" com icone Plus para abrir modal
3. Modal de cadastro: tipo (gestor/fiscal), nome, cargo, portaria_numero, data_inicio, data_fim
4. SelectInput para tipo de designacao (gestor/fiscal)
5. DataTable de designacoes: tipo, nome, cargo, portaria, periodo, ativo
6. Badge de tipo: gestor (azul), fiscal (roxo)
7. Mensagem "Selecione um contrato" quando nenhum contrato ativo

## Endpoints ou servicos envolvidos
1. **GET /api/contratos/{id}/designacoes** — lista designacoes do contrato
2. **POST /api/contratos/{id}/designacoes** — cria nova designacao com validacao
3. **PUT /api/contratos/{id}/designacoes/{desig_id}** — atualiza designacao (ativar/desativar)
4. **DELETE /api/contratos/{id}/designacoes/{desig_id}** — remove designacao
5. **Validacao backend:** verifica se mesmo nome nao e gestor e fiscal simultaneamente
6. **CRUD /api/crud/contrato-designacoes** — CRUD generico para designacoes
7. **Auth middleware** — validacao de token JWT

## Dados manipulados
1. **Designacao:** id, tipo, nome, cargo, portaria_numero, data_inicio, data_fim, ativo (boolean)
2. **Form state:** tipo ("gestor"/"fiscal"), nome, cargo, portaria_numero, data_inicio, data_fim
3. **Payload POST:** { tipo, nome, cargo, portaria_numero, data_inicio, data_fim }
4. **Tipo:** "gestor" | "fiscal" (enum restrito)
5. **Portaria:** numero do documento legal de designacao
6. **Periodo:** data_inicio e data_fim da designacao (data_fim opcional para designacao vigente)
7. **Estado UI:** designacoes (Designacao[]), showDesignacaoModal (boolean)

## Fluxo principal
1. Usuario seleciona contrato e acessa aba "Gestor/Fiscal" que carrega designacoes via GET /api/contratos/{id}/designacoes
2. Sistema exibe DataTable com designacoes existentes (tipo, nome, cargo, portaria, periodo, status ativo)
3. Usuario clica em "Nova Designacao" (icone Plus) que abre Modal com SelectInput de tipo e campos de dados
4. Usuario seleciona tipo "gestor", preenche nome, cargo, portaria_numero, data_inicio e data_fim
5. Usuario clica "Salvar" e frontend envia POST /api/contratos/{id}/designacoes com payload
6. Backend valida que mesma pessoa nao e gestor e fiscal simultaneamente (§5o), persiste designacao
7. Frontend fecha Modal, reseta formulario e chama selectContrato() que recarrega designacoes

## Fluxos alternativos
1. **FA-01 — Designar fiscal:** Usuario seleciona tipo "fiscal" e preenche dados; sistema valida unicidade
2. **FA-02 — Sem contrato selecionado:** Aba exibe mensagem "Selecione um contrato na aba Contratos"
3. **FA-03 — Desativar designacao:** Usuario marca designacao anterior como inativa; nova designacao assume
4. **FA-04 — Substituicao de gestor:** Novo gestor designado; gestor anterior e desativado automaticamente
5. **FA-05 — Sem fiscal designado:** Sistema alerta que contrato requer fiscal conforme Art. 117
6. **FA-06 — Designacao sem data fim:** data_fim nao preenchida; designacao considerada vigente indefinidamente
7. **FA-07 — Historico de designacoes:** Todas designacoes (ativas e inativas) mantidas para auditoria

## Fluxos de excecao
1. **FE-01 — Mesma pessoa como gestor e fiscal:** Backend rejeita com erro; frontend exibe mensagem "Mesma pessoa nao pode ser gestor e fiscal"
2. **FE-02 — Token expirado:** Requisicao retorna 401; sistema redireciona para LoginPage
3. **FE-03 — Nome vazio:** Campo obrigatorio nao preenchido; backend rejeita
4. **FE-04 — Portaria duplicada:** Mesmo numero de portaria ja usado; backend pode alertar
5. **FE-05 — Contrato encerrado:** Designacao em contrato nao vigente; backend pode rejeitar ou alertar
6. **FE-06 — Erro de rede:** fetch falha; try/catch captura e loga
7. **FE-07 — Conflito de datas:** data_inicio > data_fim; backend aceita mas periodo invalido
